# Database Package

## Purpose

Provides database connectivity (MariaDB, Redis), schema migration management,
plugin health tracking, and startup health checks. This is infrastructure code
used by all plugins — it does not own any business logic.

## Files

| File | Purpose |
|------|---------|
| `mariadb.go` | MariaDB connection pool setup (`NewMariaDB`) with DSN config, `DB_TLS_MODE` env var support |
| `redis.go` | Redis client setup (`NewRedis`) with connection config |
| `migrate.go` | Core migration runner using golang-migrate. Auto-runs `m.Up()` on startup. Handles dirty state with force+retry |
| `plugin_schema.go` | Plugin migration runner. Each plugin registers an `embed.FS` with numbered SQL migrations |
| `plugin_health.go` | `PluginHealthRegistry` tracks per-plugin health. Plugins degrade gracefully if their migrations fail |
| `healthcheck.go` | Comprehensive startup health check system (see below) |

## Startup Health Check System

`healthcheck.go` implements a multi-layer validation system that runs after
migrations but before the server accepts traffic. If any check fails, the
server exits with `os.Exit(1)`.

### Check Sequence

1. **Migration Version** (`checkMigrationVersion`) — Queries `schema_migrations`
   table to verify the DB is at the expected version (currently v18). Detects
   dirty state and logs instructions for manual force-retry.

2. **Critical Columns** (`checkCriticalColumns`) — Queries `information_schema.COLUMNS`
   for a configurable map of required table/column pairs. Catches schema drift
   from failed or partially-applied migrations.

3. **DB Connectivity** (`checkDBHealth`) — Pings the database with a 5-second
   timeout. Reports connection pool stats (open, in-use, idle) and warns when
   pool utilization exceeds 80%.

4. **Security Audit** (`checkSecurity`) — Detects:
   - Weak/default database passwords in production (< 12 chars or common patterns)
   - HTTP BaseURL in production (CSRF cookie vulnerability)
   - Overprivileged DB user grants (SUPER, FILE, PROCESS)
   - World-writable `schema_migrations` table (prevents attacker-side injection)

### Pre-Migration Backup

`PreMigrationBackup()` runs before `RunMigrations()`:
- Executes `mysqldump` with gzip compression
- Stores backups in a configurable directory (default: `./backups`)
- Auto-rotates old backups based on configurable retention (default: 7 days)
- Silently skips if `mysqldump` is not available on the system

### Configuration

`HealthCheckConfig` struct fields:
- `ExpectedMigrationVersion` — Target migration version to validate against
- `CriticalColumns` — `map[string][]string` of table→column pairs to verify
- `DBHost`, `DBPort`, `DBUser`, `DBPassword`, `DBName` — For backup and security checks
- `BaseURL`, `Environment` — For HTTPS and production security checks
- `BackupDir`, `BackupRetentionDays` — Backup storage configuration

### Integration

Wired in `cmd/server/main.go`:
```go
database.PreMigrationBackup(cfg)  // Before migrations
database.RunMigrations(db, cfg)   // Apply migrations
database.RunStartupHealthChecks(db, cfg)  // After migrations
```

## DB TLS Support

`NewMariaDB` reads the `DB_TLS_MODE` environment variable to configure TLS on the
MySQL DSN. Supported values: `false` (disabled, default), `true`, `skip-verify`,
`preferred`. The startup health check warns when TLS is disabled in production
environments.

## Dependencies

- **Uses:** `database/sql`, `go-sql-driver/mysql`, `golang-migrate/migrate`
- **Used by:** `cmd/server/main.go`, all plugins (via `sql.DB` and `redis.Client`)
