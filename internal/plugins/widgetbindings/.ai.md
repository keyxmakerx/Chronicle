# widgetbindings plugin

## Purpose

The foundation of the **widget-binding framework** (C-WIDGET-BINDING-P1-SPINE):
a generic, host-agnostic mapping of **host → widget-type → data-instance**. It
is the dynamic, declarative replacement for the per-widget hardcoding that
`entity_calendar`/`entity_worldstate`/`map_editor` each grew independently
(`entities.map_id` is the same idea, hardcoded for one type).

## Shape

- **`widget_bindings` table** (`migrations/001`): `(id, campaign_id, host_type,
  host_id, widget_type, instance_id, timestamps)`. **Polymorphic + FK-free** on
  `host_id` and `instance_id` — see ADR-038. `host_type` ∈ {`entity`,
  `entity_type`, `dashboard`} from day one (P1 exercises `entity` + default);
  `widget_type` is the registry slug (`calendar` in P1). Unique on
  `(campaign_id, host_type, host_id, widget_type)`.
- **`Registry`** (`registry.go`): the dynamic widget-type registry. Owning
  plugins register a `WidgetType` (`Slug` / `InstanceExists` / `DefaultInstance`
  / `ListInstances` / `CreateInstance`). It's the app-code guard for the
  `widget_type` namespace (no DB enum).
- **`Service`** (`service.go`): `Bind` / `Unbind` / `Resolve` / `OnInstanceDeleted`
  / `Sweep`. `Resolve` runs the **precedence chain**: host's own binding →
  (entity hosts) entity-type template → default (= today's behavior). Returns
  `{InstanceID, Source}` so the cascade is inspectable.
- **`Repository`** (`repository.go`): hand-written SQL; **every signature is
  campaign-scoped** so an unscoped read is unrepresentable.

## Integrity (FK-free ⇒ app-enforced — ALL THREE)

1. per-plugin **delete hook** (`OnInstanceDeleted`),
2. always-on **render-time orphan guard** (in `Resolve`, via `InstanceExists`),
3. periodic **campaign sweep** (`Sweep`).
Campaign scope is checked on **both** the host and the resolved instance
(`InstanceExists` rejects cross-campaign). MariaDB has no RLS — the app is the
only line of defense.

## Boundary

This plugin imports **no other plugin** (`HostRef`/`WidgetType` are plain
strings + interfaces). Calendar/maps/timeline depend on `widgetbindings`, never
the reverse. The calendar `WidgetType` lives in the calendar plugin
(`calendar/calendar_widget_type.go`) and is registered at app startup
(`internal/app/routes.go`).

## Footguns

- **Never add a DB FK** to `host_id`/`instance_id`, and **never rename** a
  `host_type`/`widget_type` slug — they're persisted discriminators; a rename
  orphans every stored binding (append-only namespace).
- New widget/host types are added as **registry registration / rows**, not
  schema changes — that's the modularity guarantee.
