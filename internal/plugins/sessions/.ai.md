# Sessions Plugin

Game session scheduling and RSVP tracking for campaigns. Sessions represent
real-world play sessions (game nights) with attendee management and entity linking.

## Architecture

- **model.go**: Session, Attendee, SessionEntity structs + DTOs.
  Status constants: `planned`, `completed`, `cancelled`.
  RSVP constants: `invited`, `accepted`, `declined`, `tentative`.
  Entity role constants: `mentioned`, `encountered`, `key`.
- **repository.go**: SQL CRUD for sessions + attendees + entity linking.
  ListByCampaign sorts planned first, then by scheduled_date. AddAttendee
  uses ON DUPLICATE KEY UPDATE for upsert.
- **service.go**: Validation, CRUD orchestration, InviteAll for batch invites,
  RSVP status updates, entity linking with role validation.
- **handler.go**: HTTP handlers for list, detail, create, update, delete,
  RSVP, entity link/unlink. MemberLister interface for cross-plugin
  dependency (auto-invite campaign members). HTMX-aware responses.
- **routes.go**: Route registration with role middleware.
- **sessions.templ**: List page with status grouping (Upcoming/Completed/Cancelled),
  detail page with attendee list and RSVP buttons, create modal, edit modal
  (`editSessionModal` — pre-populated fields, JSON PUT to `UpdateSessionAPI`).

## Data Model

- `sessions` — id, campaign_id, name, summary, notes, notes_html,
  scheduled_date, calendar_year/month/day, status, sort_order,
  created_by, created_at, updated_at
- `session_attendees` — id, session_id, user_id, status (invited/accepted/
  declined/tentative), responded_at
- `session_entities` — id, session_id, entity_id, role (mentioned/
  encountered/key)
- `member_availability` (migration 002, C-SCHED-P1) — id, campaign_id, user_id,
  day_of_week (0=Sun..6=Sat), start_minute, end_minute, state
  (available/preferred), tz (IANA), updated_at. RECURRING weekly pattern stored
  as ZONE-LOCAL wall-clock, never UTC (RC-12.5). Absence of a row = unavailable.
- `availability_exceptions` (migration 002) — id, campaign_id, user_id, on_date,
  start_minute, end_minute, state, tz, updated_at. Per-date override; any row for
  a date fully replaces the recurring pattern for that date.

Availability lives in its OWN tables — deliberately NOT `session_attendees` —
so it stays out of campaign/AI export egress by construction (design §5;
pinned by `availability_egress_test.go`).

## Cross-Plugin Dependencies

- **campaigns**: MemberLister interface provides ListMembers() for auto-inviting
  all campaign members when a session is created.
- **addons**: Registered as addon slug "sessions" — must be enabled per campaign.

DM-granted inside the handler by role (design §5), not by a separate route.

## RSVP Flow

1. Session created → all campaign members auto-invited (status: `invited`)
2. Players see session with RSVP buttons (Going / Maybe / Can't Make It)
3. RSVP updates via HTMX POST, returns refreshed attendee list fragment
4. Session detail shows attendee count summary (e.g., "3/5 going")

## Session Edit UI

- **Edit button**: Pencil icon in session detail header, visible to Scribe+ users.
  Opens `#edit-session-modal` dialog.
- **Edit modal** (`editSessionModal` templ component): Pre-populated with current
  session values (name, scheduled_date, summary, status, recurrence settings).
  Uses `<dialog>` element matching `createSessionModal` pattern.
- **Status select**: Dropdown with planned/completed/cancelled. Uses `data-initial`
  attribute + JS initialization (templ doesn't support `selected?=` syntax).
- **Submit**: JS serializes form to JSON, sends `PUT` to existing `UpdateSessionAPI`
  endpoint via `Chronicle.apiFetch()`, reloads page on success.
- **Recurrence**: AlpineJS `x-data` toggle for recurring settings (same as create modal).

## Key Design Decisions

- **Separate from calendar events**: Sessions are their own entity, not calendar
  events. This keeps the calendar clean for in-world events while sessions
  track real-world scheduling.
- **Auto-invite on create**: All campaign members are automatically invited when
  a session is created, reducing manual setup.
- **IDOR protection**: `requireSessionInCampaign()` verifies session belongs to
  the campaign in the URL path before any operation.
- **Upsert attendees**: AddAttendee uses ON DUPLICATE KEY UPDATE so re-inviting
  or changing RSVP is idempotent.

## Availability Scheduler (C-SCHED-P1, Slice 1)

Real-world session scheduling: members paint a recurring weekly availability
grid; the DM sees everyone stacked as a translucent heatmap to find the best
slots. Composes with per-user IANA timezones. Proposals/notifications are P2/P3.

- **Files**: `availability_model.go` (types, DTOs, palette), `availability_repository.go`
  (CRUD on `member_availability` + `availability_exceptions`, methods on the
  existing `sessionRepository`), `availability_service.go` (validation + overlay
  orchestration), `availability_overlay.go` (the PURE DST-correct projection),
  `availability_handler.go` (HTTP + role gating), `availability.templ` (page
  shell), `static/js/availability.js` (paint grid + heatmap rendering).
- **DST correctness** is the crux and lives in `internal/timeutil` (stdlib-only,
  no plugin imports — shared with the real-calendar lane): a recurring block is a
  zone-local wall-clock resolved to an absolute instant only against a concrete
  week's real dates (`time.Date(y,m,d,0,min,0,0,loc)`), then converted to the
  viewer's zone. `buildWeekOverlay` projects each member's pattern onto the
  target week in the DM's zone. Tested with cross-zone + spring-forward cases.
- **Signed encoding** (`mockups/availability-calendar-concept.html`): density
  underlay = the heatmap (steep green ramp, deeper = more free), count gutter,
  ★ = full house AND all prefer, slim OPAQUE per-member lanes so identity never
  blends into the wash, explicit DM-zone label, WCAG-AA.
- **Security**: member-only (authed group, never public); per-member detail
  gated to owner / DM-granted; own-tables keep availability out of export egress
  (`availability_egress_test.go`).
- **Deferred to later slices**: slot proposals, per-option RSVP, notifications,
  the month view, and a richer per-date exception editor UI.
