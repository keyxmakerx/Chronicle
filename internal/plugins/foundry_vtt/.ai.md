# foundry_vtt Sub-Plugin

The `foundry_vtt` plugin is the Foundry-VTT-specific extension to the
generic `packages` plugin. Every Foundry assumption Chronicle used to
make natively now lives here: per-campaign signed manifest URLs,
per-campaign pinning, the `chronicle-package.json` descriptor reader,
and the post-install `module.json` version rewrite that fixes the
operator's "Foundry shows v0.1.0 after I installed v0.2.0" bug.

The `packages` plugin remains generic — zero Foundry-specific code
in `packages/`. This plugin is the only place strings like
`"manifest"`, `"download"`, `"module.json"`, and
`"chronicle-package.json"` appear in the Chronicle backend.

## Why this plugin exists

The operator's directive, verbatim: *"This should be very modular and
basically the foundry package would instruct these processes, not
baked in directly to chronicle. We don't want foundry stuff inside
chronicle natively, because then that just makes a redundant mess."*

Three rounds of architectural drift left two parallel Foundry-module
systems coexisting in `packages/` and the (now-deleted-in-C-FMC-5c)
`foundry_modules/`. The C-FMC-5 consolidation collapses them: this
sub-plugin owns ALL Foundry-specific behavior, `packages` owns
generic external-repo management. The boundary is the
`packages.PostInstallHook` extension point added in C-FMC-5a.

## Modularity contract

**This plugin is INERT when no foundry-module typed package is
registered in the `packages` catalog.** The routes mount
unconditionally at boot, but:

- The owner tab fragment renders an empty state ("ask an admin to
  register the module").
- The public manifest endpoint returns a categorized 503 with
  `code=no_package_registered` and an actionable message.
- The `PostInstallHook` is registered but never fires (no
  foundry-module installs to fire on).

Translation: removing the Chronicle-Foundry-Module package from
`/admin/packages` turns this plugin off. There's nothing else to
configure. That's the modularity goal.

## Architectural decisions

Logged here per the operator's explicit ask — *"this initiative has
had three architectural misses partly because prior decisions
weren't documented. Don't repeat that."*

### Why a sub-plugin and not a `packages` extension

**Decision**: Build `foundry_vtt` as a sibling plugin alongside
`packages`, communicating via the `PostInstallHook` interface and
the `PackageReader` adapter.

**Rejected**: Extend `packages` directly with Foundry-aware methods.
The operator explicitly disallowed this — it's exactly the
"Foundry stuff baked in" pattern that produced the redundant mess.

**Trade-off**: One extra plugin in the codebase. Cost is low; the
boundary is sharp; future TTRPG VTT extensions (Roll20, Owlbear)
follow the same pattern.

### Why descriptor-driven (and not hardcoded)

**Decision**: Read `chronicle-package.json` from the extracted
install dir. The descriptor declares the manifest path, which
fields to rewrite, and the endpoint URL templates. Hardcoded
defaults apply when no descriptor is present (so old zips keep
working).

**Rejected**: Hardcode every Foundry assumption (manifest path
`"module.json"`, rewrite fields `["manifest", "download"]`, URL
templates) inside Chronicle. Future Foundry module variants —
nested layouts, additional rewrite fields, Foundry-system rather
than Foundry-module packages — would each need a Chronicle code
change.

**Trade-off**: Schema versioning surface. Mitigation: hook fails
loudly on unrecognized major version; minor additions are
backward-compatible (older Chronicle ignores unknown fields).

### Why per-campaign signed URLs (and not session auth)

**Decision**: Mint a per-campaign install URL with an HMAC-signed
token. The token bakes in the campaign's current `token_version`
counter; rotation = bump the counter, every previously-issued URL
stops verifying.

**Rejected**: Bearer-token auth on the manifest endpoint. Foundry
doesn't support custom auth headers on module installation —
it polls the URL with no session context, so the URL itself is
the credential.

**Trade-off**: URL is bearer-equivalent (anyone who copies it can
install). Mitigated by rotation: when a campaign owner suspects a
leak, one click invalidates every existing URL.

### Why the hook reads `chronicle-package.json` with fallback

**Decision**: Missing descriptor = apply hardcoded defaults
equivalent to the canonical Foundry module's published descriptor.
Present-but-invalid descriptor = fail the install loudly.

**Rejected option A**: Require every module to ship a descriptor.
Would break every existing module zip until FM-PKG-DESCRIPTOR
ships and all modules re-cut releases.

**Rejected option B**: Silently fall back even when descriptor is
present-but-invalid. Would mask upstream packaging bugs for hours
or days. Operator's fail-loud contract from C-FMC-5a applies.

**Trade-off**: Two code paths to test (descriptor + fallback).
Tests pin both — see `descriptor_test.go` +
`post_install_hook_test.go`.

### Why tokens live in a separate table (not CampaignSettings JSON)

**Decision**: `foundry_module_campaign_tokens` is a dedicated table
with `(campaign_id, token_version, rotated_at)`. The
`FoundryModulePin` string lives on `CampaignSettings` (campaigns
plugin); tokens live here.

**Rejected**: Embed `token_version` in `CampaignSettings` JSON.
Token rotation would need a campaigns-plugin write path for what's
fundamentally a foundry_vtt operation. The table is cheap and
keeps the responsibility lines clean.

**Trade-off**: Two storage layers for related state. Mitigated by
the narrow `CampaignSettingsAdapter` interface — this plugin only
sees the three methods it needs.

### Why the table stays named `foundry_module_campaign_tokens` in C-FMC-5b

**Decision**: Continue reading the existing
`foundry_module_campaign_tokens` table in this PR. The rename to
`foundry_vtt_campaign_tokens` happens in C-FMC-5c alongside the
`foundry_modules` plugin deletion.

**Rejected**: Rename in C-FMC-5b. Would require synchronously
updating `foundry_modules`' SQL queries (since both plugins
coexist during the parallel period) — mechanical but increases
this PR's scope without buying anything. The rename naturally
pairs with `foundry_modules` deletion in C-FMC-5c.

**Deviation note**: The operator's C-FMC-5b dispatch suggested
renaming in this PR; deferred per the trade-off above. Tracked in
the PR description's "Deviations" section.

## Files

| File | Purpose |
|---|---|
| `errors.go` | Categorized `Error` type + pre-defined constructors. Every public error follows the four-clause format (what / detail / cause / action) the operator requires. |
| `model.go` | `CampaignToken`, `OwnerTabData`, `PackageDescriptor` types. Pure data, no behavior. |
| `token.go` | HMAC-SHA256 token signer. Ported verbatim from `foundry_modules/token.go` with the domain prefix changed to `foundry-vtt:` for cross-plugin replay defense. |
| `descriptor.go` | Reads `chronicle-package.json` from the install dir. `loadDescriptor()` returns defaults + sentinel on missing file; `*Error` on present-but-invalid. |
| `repository.go` | CRUD for the per-campaign token table. Three methods: Get / Upsert / BumpVersion. |
| `service.go` | Pin resolver, manifest builder, token verifier, owner tab data assembler. Depends on `PackageReader` (packages plugin) and `CampaignSettingsAdapter` (campaigns plugin) via interfaces — no direct imports of those plugins' types beyond the adapter shapes. |
| `post_install_hook.go` | Implements `packages.PostInstallHook` for `PackageTypeFoundryModule`. Reads descriptor → rewrites `module.json` version at the declared path. |
| `handler.go` | Echo handlers + the `respondError` mapper that turns `*Error` into the categorized JSON response Foundry's FM-CSU-DIAG parses. Public + owner endpoints. |
| `routes.go` | `RegisterOwnerRoutes` + `RegisterPublicRoutes`. URL shapes locked at `/foundry-vtt/...` (owner) and `/api/v1/campaigns/:cid/foundry-vtt/...` (public). |
| `owner_tab.templ` | Owner settings tab fragment. HTMX-served at `/campaigns/:id/foundry-vtt/settings-tab`. Renders empty state when no foundry-module package is registered. |

## Database tables

### `foundry_module_campaign_tokens` (read during C-FMC-5b parallel period)

```
campaign_id   CHAR(36)  NOT NULL  PRIMARY KEY
token_version INT       NOT NULL  DEFAULT 1
rotated_at    TIMESTAMP NOT NULL  DEFAULT CURRENT_TIMESTAMP
FOREIGN KEY (campaign_id) REFERENCES campaigns(id) ON DELETE CASCADE
```

`token_version` is the rotation counter. Sign payload =
`HMAC(secret, "foundry-vtt:{campaign_id}:{token_version}")`. Rotate =
increment counter, every previously-signed URL becomes invalid on
next manifest request.

Table is created by `foundry_modules/migrations/001_*.up.sql`.
Renamed to `foundry_vtt_campaign_tokens` in C-FMC-5c when
`foundry_modules` is deleted.

## Extension point: PostInstallHook

`foundry_vtt.NewPostInstallHook()` returns a struct satisfying
`packages.PostInstallHook` (added in C-FMC-5a). Registered at boot
via `packages.RegisterPostInstallHook(pkgService, hook)`.

Fires after every `PackageTypeFoundryModule` install:

1. Loads `chronicle-package.json` from the install dir (or applies
   defaults).
2. Reads `module.json` at the descriptor-declared `moduleJsonPath`.
3. Overwrites the `version` field with the installed version
   string.
4. Writes the file back.

Errors fail the install loudly (per C-FMC-5a contract). The
packages plugin's `InstallVersion` cleans up `destDir` on hook
failure, so a retry installs from scratch.

## Extension point: chronicle-package.json descriptor

The schema is owned on the Foundry side
(FM-PKG-DESCRIPTOR — `keyxmakerx/Chronicle-Foundry-Module#35`).
Chronicle's adoption is described here.

### Schema v1 (canonical)

```jsonc
{
  "schemaVersion": 1,
  "package": {
    "id": "chronicle-sync",
    "kind": "foundry-module",
    "moduleJsonPath": "module.json"
  },
  "serving": {
    "rewriteFields": ["manifest", "download"],
    "manifestEndpoint": "/api/v1/campaigns/{campaign_id}/foundry-vtt/module.json?token={token}",
    "downloadEndpoint": "/api/v1/campaigns/{campaign_id}/foundry-vtt/module.zip?token={token}",
    "perCampaignSignedToken": true,
    "zipContentRoot": ""
  }
}
```

### Fallback semantics

When no `chronicle-package.json` exists in the install dir,
`defaultDescriptor()` applies. The defaults match the canonical
schema above — equivalent to what the Foundry module's published
descriptor declares.

This means C-FMC-5b can deploy BEFORE FM-PKG-DESCRIPTOR ships —
old zips just use the fallback, new zips drive behavior from the
descriptor.

### Schema versioning

`schemaVersion` is the load-bearing field. The hook recognizes
only `currentSchemaVersion` (1 today). Unknown major versions
fail the install loudly. Future v2+ requires coordinated Foundry +
Chronicle releases.

Within v1: missing optional fields fall back to defaults via
`applyDefaults()`. Modules can ship minimal descriptors (just
`schemaVersion` + `package.id`) and inherit everything else.

## Error handling contract

Every public-facing error follows this format:

```
<what failed>: <technical detail>. <likely cause>. <actionable next step>.
```

See `errors.go` for all pre-defined constructors. Examples:

| Constructor | Category | HTTP Status | Format-compliant Message |
|---|---|---|---|
| `ErrInvalidToken` | auth | 403 | Wrong HMAC or rotated → operator rotates URL via campaign settings. |
| `ErrNoPackageRegistered` | config | 503 | No foundry-module package → admin adds it via `/admin/packages`. |
| `ErrPinnedVersionNotInstalled` | config | 503 | Pin to missing version → admin installs OR owner changes pin. |
| `ErrNoVersionAvailable` | config | 503 | Package exists, no version installed → admin clicks Install. |
| `ErrTokenNotInitialized` | config | 503 | First-time campaign hit before owner visited settings → owner opens VTT Setup Guides. |
| `ErrDescriptorInvalid` | validation | 422 | Bad descriptor in zip → fix upstream OR remove descriptor for defaults. |
| `ErrModuleJSONMissing` | internal | 500 | Zip malformed → admin retries OR upstream fix. |
| `ErrCampaignNotFound` | not_found | 404 | URL refers to deleted/wrong-instance campaign → owner mints a fresh URL. |

JSON shape returned to public endpoints (Foundry's FM-CSU-DIAG
consumes this):

```json
{
  "error": "<code>",
  "message": "<format-compliant message>",
  "category": "<config|auth|not_found|validation|internal>"
}
```

## C-FMC-5b scope notes

This PR deliberately ships LESS than the C-FMC-5b dispatch
described. Deviations documented:

1. **No admin section.** The dispatch called for a "campaigns
   using v0.1.5" expandable cards UI in `/admin/packages`.
   Deferred — operator can use `foundry_modules`' existing admin
   UI during the parallel period. The cards rebuild on the
   renamed table in C-FMC-5c.
2. **No notify / force-pin handlers.** Same reasoning — those
   admin actions live in `foundry_modules` for the parallel
   period. Re-implemented in C-FMC-5c.
3. **No migration in this PR.** Token table rename is deferred to
   C-FMC-5c alongside `foundry_modules` deletion. See decision
   log §"Why the table stays named ..." above.

This shrinks the PR to the load-bearing minimum: hook +
descriptor + per-campaign endpoints + owner tab. The architecture
+ error contract is fully in place; remaining surface is
mechanical port-and-delete in C-FMC-5c.
