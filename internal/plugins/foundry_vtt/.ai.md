# foundry_vtt Sub-Plugin

The `foundry_vtt` plugin is the Foundry-VTT-specific extension to the
generic `packages` plugin. Every Foundry assumption Chronicle used to
make natively now lives here: per-campaign signed manifest URLs,
per-campaign pinning, the `chronicle-package.json` descriptor reader,
and the post-install `module.json` version rewrite that fixes the
operator's "Foundry shows v0.1.0 after I installed v0.2.0" bug.

The `packages` plugin remains generic — zero Foundry-specific code
in `packages/`. This plugin is the only place strings like
`"manifest"`, `"download"`, `"module.json"`, and
`"chronicle-package.json"` appear in the Chronicle backend.

## Why this plugin exists

The operator's directive, verbatim: *"This should be very modular and
basically the foundry package would instruct these processes, not
baked in directly to chronicle. We don't want foundry stuff inside
chronicle natively, because then that just makes a redundant mess."*

Three rounds of architectural drift left two parallel Foundry-module
systems coexisting in `packages/` and the (now-deleted-in-C-FMC-5c)
`foundry_modules/`. The C-FMC-5 consolidation collapses them: this
sub-plugin owns ALL Foundry-specific behavior, `packages` owns
generic external-repo management. The boundary is the
`packages.PostInstallHook` extension point added in C-FMC-5a.

## Modularity contract

**This plugin is INERT when no foundry-module typed package is
registered in the `packages` catalog.** The routes mount
unconditionally at boot, but:

- The owner tab fragment renders an empty state ("ask an admin to
  register the module").
- The public manifest endpoint returns a categorized 503 with
  `code=no_package_registered` and an actionable message.
- The `PostInstallHook` is registered but never fires (no
  foundry-module installs to fire on).

Translation: removing the Chronicle-Foundry-Module package from
`/admin/packages` turns this plugin off. There's nothing else to
configure. That's the modularity goal.

## Architectural decisions

Logged here per the operator's explicit ask — *"this initiative has
had three architectural misses partly because prior decisions
weren't documented. Don't repeat that."*

### Why a sub-plugin and not a `packages` extension

**Decision**: Build `foundry_vtt` as a sibling plugin alongside
`packages`, communicating via the `PostInstallHook` interface and
the `PackageReader` adapter.

**Rejected**: Extend `packages` directly with Foundry-aware methods.
The operator explicitly disallowed this — it's exactly the
"Foundry stuff baked in" pattern that produced the redundant mess.

**Trade-off**: One extra plugin in the codebase. Cost is low; the
boundary is sharp; future TTRPG VTT extensions (Roll20, Owlbear)
follow the same pattern.

### Why descriptor-driven (and not hardcoded)

**Decision**: Read `chronicle-package.json` from the extracted
install dir. The descriptor declares the manifest path, which
fields to rewrite, and the endpoint URL templates. Hardcoded
defaults apply when no descriptor is present (so old zips keep
working).

**Rejected**: Hardcode every Foundry assumption (manifest path
`"module.json"`, rewrite fields `["manifest", "download"]`, URL
templates) inside Chronicle. Future Foundry module variants —
nested layouts, additional rewrite fields, Foundry-system rather
than Foundry-module packages — would each need a Chronicle code
change.

**Trade-off**: Schema versioning surface. Mitigation: hook fails
loudly on unrecognized major version; minor additions are
backward-compatible (older Chronicle ignores unknown fields).

### Why per-campaign signed URLs (and not session auth)

**Decision**: Mint a per-campaign install URL with an HMAC-signed
token. The token bakes in the campaign's current `token_version`
counter; rotation = bump the counter, every previously-issued URL
stops verifying.

**Rejected**: Bearer-token auth on the manifest endpoint. Foundry
doesn't support custom auth headers on module installation —
it polls the URL with no session context, so the URL itself is
the credential.

**Trade-off**: URL is bearer-equivalent (anyone who copies it can
install). Mitigated by rotation: when a campaign owner suspects a
leak, one click invalidates every existing URL.

### loadDescriptor canonical-vs-fallback pin (C-SEC-4-AMENDED)

The fallback `defaultDescriptor()` values (`descriptor.go:142-156`) MUST match what the canonical `chronicle-package.json` ships from `chronicle-foundry-module`. Otherwise an install WITH a descriptor would behave differently from an install WITHOUT one, defeating the descriptor's "explicitly declare what would otherwise be implicit" purpose.

`descriptor_fallback_test.go` + `testdata/chronicle-package.json` pin this alignment as a committed snapshot. Per-field assertions; `package.id` excluded (descriptor-required, intentionally empty in defaults). The snapshot is NOT a live fetch — Chronicle's CI doesn't reach into Foundry-Module on every build. The Foundry-Module-side comment in `tools/check-package-descriptor.mjs` (`keyxmakerx/chronicle-foundry-module`, cross-pushed Cordinator-side as part of the SEC-4-AMENDED recovery) instructs maintainers to regenerate this snapshot whenever they change the canonical.

Full canonical-vs-fallback table + the `applyDefaults` `PerCampaignSignedToken` always-override edge case + snapshot-limitation tradeoff are documented in `cordinator/decisions/2026-05-22-loadDescriptor-fallback.md`.

### Why the hook reads `chronicle-package.json` with fallback

**Decision**: Missing descriptor = apply hardcoded defaults
equivalent to the canonical Foundry module's published descriptor.
Present-but-invalid descriptor = fail the install loudly.

**Rejected option A**: Require every module to ship a descriptor.
Would break every existing module zip until FM-PKG-DESCRIPTOR
ships and all modules re-cut releases.

**Rejected option B**: Silently fall back even when descriptor is
present-but-invalid. Would mask upstream packaging bugs for hours
or days. Operator's fail-loud contract from C-FMC-5a applies.

**Trade-off**: Two code paths to test (descriptor + fallback).
Tests pin both — see `descriptor_test.go` +
`post_install_hook_test.go`.

### Why tokens live in a separate table (not CampaignSettings JSON)

**Decision**: `foundry_module_campaign_tokens` is a dedicated table
with `(campaign_id, token_version, rotated_at)`. The
`FoundryModulePin` string lives on `CampaignSettings` (campaigns
plugin); tokens live here.

**Rejected**: Embed `token_version` in `CampaignSettings` JSON.
Token rotation would need a campaigns-plugin write path for what's
fundamentally a foundry_vtt operation. The table is cheap and
keeps the responsibility lines clean.

**Trade-off**: Two storage layers for related state. Mitigated by
the narrow `CampaignSettingsAdapter` interface — this plugin only
sees the three methods it needs.

### Why the token table rename was deferred to C-FMC-5c

**Decision**: C-FMC-5b read the existing
`foundry_module_campaign_tokens` table (left over from
foundry_modules). C-FMC-5c does the rename to
`foundry_vtt_campaign_tokens` alongside the `foundry_modules` plugin
deletion.

**Rejected** (C-FMC-5b path): rename in C-FMC-5b. Would have required
synchronously updating `foundry_modules`' SQL queries while both
plugins coexisted — mechanical but increases that PR's scope.

The rename paired naturally with `foundry_modules` deletion in
C-FMC-5c. Token row data is preserved by `RENAME TABLE` — existing
campaigns keep their token versions across the rename.

### C-FMC-9 post-install UX polish + owner-tab blank-page fix + force-pin error wrap

Four bugs surfaced in operator's testing of the consolidated C-FMC-7/8 work; addressed in one PR because the fixes share verification (you can't confirm the older-campaigns prompt works without confirming force-pin actually writes; can't confirm owners see the right thing without confirming the banner can navigate there).

**Bug 1 — post-install "older campaigns" prompt** doesn't auto-trigger. Fixed by adding a **"Show affected campaigns"** call-to-action button to the autopin install banner (`autopin_banner.templ`). New inline-IIFE helper `showAffectedCampaignsOnClick(previousVersion)` in `onclick_handlers.go` locates the version-scoped Campaigns expand button (added stable ID `fvtt-campaigns-trigger-{sanitized-version}` to `packages/packages.templ`), scrolls it into view, and programmatically clicks it.

Architectural choice: **no new HX-Trigger event-bus mechanism**. The dispatch suggested a packages-plugin response header that foundry_vtt listens for. That would have required either Alpine or document.addEventListener — both forbidden by the C-FMC-8 inline-IIFE rule. The banner button approach uses pure DOM manipulation in a single inline IIFE, preserving the C-FMC-5c modularity boundary (packages stays foundry-agnostic; the cross-plugin coupling is the stable button-ID contract, mirrored on both sides via `sanitizeForID` (packages) and `sanitizeVersionForDOMID` (foundry_vtt). Both helpers have contract tests pinning their alignment.

**Bug 2 — force-pin "succeeds but no DB write"**. **Code inspection of `ForcePinCampaign` → `SetPinnedVersion` → `CampaignSettingsAdapter.SetFoundryModulePin` → `campaignService.SetFoundryModulePin` → `repo.UpdateSettings` shows the error chain is correctly propagated end-to-end.** No silent swallow exists on the code path. The operator-reported symptom is therefore most likely a UI-refresh issue (page reloads but the Campaigns expand collapses, so visible state appears unchanged), not a data-persistence issue. The visual-refresh issue is addressed by Bug 1's "Show affected campaigns" CTA.

Defensive C-FMC-9 still wraps the error in `ForcePinCampaign` with explicit context (`force-pin campaign %q to %q: %w`) so audit logs make the failed code path obvious. The `%w` preserves the typed error chain for `errors.As(err, &fe)` so the handler's categorized JSON response stays accurate. Test: `cfmc9_test.go::TestForcePinCampaign_ErrorWrapsAndUnwraps`.

**Bug 3 — owner settings page renders blank**. Fixed by:

1. `handler.go::OwnerTabFragmentHandler` no longer returns `apperror.NewMissingContext()` / `apperror.NewInternal(...)`. The handler ALWAYS returns 200 with rendered content; error cases are surfaced via a new `OwnerTabErrorState(message, action)` templ component inside the same swap target. HTMX never sees a 4xx/5xx response, so the loading spinner is always replaced with either the success fragment or an inline error card.
2. `campaigns/settings.templ`'s hx-get block gains an `hx-on::response-error` fallback that replaces the spinner with an inline error message on network-level failures (the handler itself always returns 200 now, so this only triggers on connection failures).

**Bug 4 — UI polish**:
- Admin "Campaigns Pinned to vX" cards: now genuine `.card` elements with explicit borders + a prominent amber callout when older-version usage exists (was: bare flex divs, no callout).
- Owner tab restructured for install-URL prominence: install URL is the focal-point card (accent border, bigger Copy button), current-version + pin selector are secondary, player instructions are wrapped in a `<details>` disclosure (collapsed by default for owners who don't need them).

### C-FMC-8 inline-IIFE onclick handlers + deferred admin banner

**Decision**: every onclick handler inside an HTMX-swapped fragment
in this plugin (admin.templ, owner_tab.templ, autopin_banner.templ)
uses an INLINE IIFE in the attribute value, NOT a templ `script`
helper. The JS body is built in Go as a string in
`onclick_handlers.go`, wrapped in a `templ.ComponentScript` with
empty `Function` (no `<script>` tag) and the body in `Call`. Templ
writes Call directly into the `onclick="..."` attribute.

**Rejected (C-FMC-5b/5c approach, now overturned)**: templ `script`
helpers. They emit `<script>function __templ_X(){...}</script>`
alongside the button + an onclick that references `__templ_X(...)`
by name. When the fragment loads via hx-get and innerHTML-swap,
browsers do NOT reliably execute the `<script>` tag synchronously
with the surrounding HTML. The button can be clicked before the
function is defined, producing:

```
Uncaught ReferenceError: __templ_X is not defined
```

PRs #302, #303, C-FMC-5c (admin Force-update/Notify buttons), and
C-FMC-6 (owner Save Pin + Rotate Token) all hit this. Prior
versions of admin.templ contained a doc comment claiming templ
scripts were "HTMX-swap-safe"; that claim was load-bearing wrong
and was struck in C-FMC-8.

**The inline-IIFE pattern**: JS body wrapped in `(function(){...})()`,
single-quoted JS strings throughout (double quotes would close the
outer `onclick="..."` since templ doesn't HTML-escape Call),
JS-escaped interpolated values via `text/template.JSEscapeString`
so embedded apostrophes/backslashes can't break out.

**Contract tests** in `onclick_handlers_test.go`: every handler is
asserted to (a) start with `(function(`, (b) contain no literal
`"` character, (c) contain no `__templ_` substring. Regression
fails loudly.

**Deferred admin banner shipped here**: C-FMC-6 deferred the
dismissible admin banner for the auto-pin install summary, on the
reasoning that the security events dashboard was a sufficient audit
trail. C-FMC-8 ships the banner: foundry_vtt's `AutoPinOnInstall`
persists the latest summary to `site_settings.foundry_vtt.latest_autopin_summary`;
a new `/admin/foundry-vtt/autopin-banner` endpoint reads the summary
+ dismissal timestamp and returns the banner HTML or empty;
`/admin/foundry-vtt/autopin-banner/dismiss` stamps a Unix timestamp
that silences the banner until a newer summary arrives. The packages
plugin's `packages.templ` embeds the fragment via hx-get on
`/admin/packages` load — packages stays foundry-agnostic.

### Coordinator-level pre-merge rules (C-FMC-8 codification)

After three production bugs from cross-cutting architectural
changes, the operator added these load-bearing checks to every
future Chronicle PR. Re-listed here so they don't get lost:

1. **Fragment consumer trace**: every PR creating a fragment
   served via hx-get lists every consumer (the templ files that
   embed it). Every PR deleting a fragment endpoint verifies all
   consumers were updated to consume something else. Origin:
   C-FMC-6 owner_tab wire-up gap.

2. **On-disk artifact trace**: every PR changing how a URL gets
   served verifies the on-disk artifact (extracted zip's
   module.json, cached files, etc.) carries the new URL. Origin:
   C-FMC-7 zip-rewrite gap — serve-time rewriting wasn't enough
   because Foundry reads the on-disk module.json after install.

3. **HTMX click-handler trace**: every PR adding interactive
   elements to an HTMX-loaded fragment verifies the click handler
   is an inline IIFE (via onclick_handlers.go-style Go-side
   builders returning ComponentScript with empty Function), NOT a
   templ `script` helper or a delegated `document.addEventListener`.
   Origin: C-FMC-8 ReferenceError bugs.

These rules are also encoded as contract tests where applicable
(see `onclick_handlers_test.go`, `zip_rewrite_test.go`,
`auto_pin_test.go`). Future contributors who violate them get a
failing test, not a production bug.

### C-FMC-7 per-campaign zip rewriting at download time

**Decision**: The download endpoint streams a zip with the embedded
`module.json` REWRITTEN to carry per-campaign Chronicle URLs.
Every download walks the install dir, swaps the descriptor-declared
`module.json` for the rewritten bytes (the same bytes the manifest
endpoint serves), and copies every other file byte-for-byte.

**Rejected (C-FMC-5c reasoning, now overturned)**: serve raw on-disk
zip and rely on serve-time manifest rewriting at the manifest
endpoint. The C-FMC-5c plan deleted `repackFoundryZip` (which used
to do install-time URL baking for the old server-wide model) with
the reasoning: *"new per-campaign manifest endpoint rewrites URLs
at serve-time, no cached zip to repack."* That reasoning had a
hole: it fixed only the FIRST manifest fetch. After Foundry extracts
the zip and reads the on-disk `module.json`, that file becomes
Foundry's source-of-truth for update-check URLs — and it carried
the upstream GitHub URLs unchanged. Update checks reverted to
GitHub forever.

**The architectural correction**: per-campaign URLs require per-
campaign zips. We can't bake URLs into a single cached zip (different
campaigns get different URLs); we can't omit the rewrite (Foundry
needs Chronicle URLs in the on-disk file). The only viable path is
streaming a freshly-rewritten zip per download. No caching — each
request mints a fresh signed token, the zip's module.json carries
that token, the file is small enough that re-zipping per request
is cheap.

**Rejected (alternative)**: cache rewritten zips keyed on
`(campaign_id, token_version)`. Would speed up repeated downloads
from the same campaign at the cost of cache invalidation
complexity (every token rotation requires invalidating that
campaign's cached zip; every new install requires invalidating
every campaign's cache for the previous version). The streaming
path is simpler + performance overhead is negligible: zips are
<5 MB, downloads are infrequent (per Foundry install + occasional
re-installs).

**Implementation note**: the manifest endpoint and download endpoint
share `resolveCampaignManifest` as their single source of truth for
descriptor loading + URL rewriting. Both produce identical rewritten
bytes for the same `(campaign_id, token_version)` pair, so the
manifest endpoint's response and the zip's embedded `module.json`
stay consistent.

### Why the wire-up gap happened (C-FMC-5b/5c) and how C-FMC-6 fixed it

**What happened**: C-FMC-5b created `owner_tab.templ` at
`/campaigns/:id/foundry-vtt/settings-tab` as a complete replacement
for the old foundry_modules tab fragment. C-FMC-5c deleted the
foundry_modules plugin including its `/campaigns/:id/foundry/settings-tab`
route. But neither PR updated `campaigns/settings.templ`, which kept
its `hx-get` pointed at the old (now-404) URL. Owners loading
campaign settings saw the loader spinner stall instead of the
install URL surface. Operator couldn't get a manifest URL → couldn't
install Foundry → blocked on testing.

**Why it slipped past review**: both PRs verified their own plugins'
internal consistency — the new fragment served correctly; the old
plugin deleted cleanly. Neither PR explicitly verified the
*consumer* (campaigns/settings.templ) was updated. Cross-plugin
wire-up is a known seam in this codebase (sub-plugins extend pages
owned by other plugins via hx-get) and the gap fell into the seam.

**The fix (C-FMC-6 Part A)**: one URL change in settings.templ +
load-bearing code comments at the embed point explaining the
sub-plugin pattern so future readers know the relationship without
having to discover it via bug reports.

**Forward-looking discipline**: every PR that creates a fragment
served from one plugin and consumed via hx-get by another must
explicitly verify the consumer in its test plan. Every PR that
deletes a fragment endpoint must verify all consumers were updated
to consume something else. The operator added this as a coordinator-
level check for future dispatches.

### C-FMC-6 auto-pin-on-install (vs. silent forward-bump)

**Decision**: After every foundry-module install, every campaign with
an empty pin (auto-tracking) gets explicit-pinned to the PREVIOUS
version. The admin sees the version spread via the C-FMC-5c
"Campaigns Using v0.X.Y" UI and decides per-campaign whether to bump.

**Rejected option A**: silent forward-bump (the C-FMC-5b/5c
behavior). Auto-tracking campaigns silently follow the latest
installed version. Operator hit this in production: installed
v0.1.11 over v0.1.10, both auto-tracking campaigns instantly jumped
without notification.

**Rejected option B**: block the install until admin confirms the
spread. Adds a modal/confirmation flow to the install button.
Disruptive; admin's already in the "I want to install" mindset by
the time they click. Auto-pin-then-notify lets the install complete
+ surfaces the version spread for review.

**Trade-off**: campaigns that *expected* to track latest now require
explicit pin updates. Mitigated by the C-FMC-5c admin "Force-update
All Older" mass action — one click moves every older-pinned campaign
to the latest.

### C-FMC-6 one-time auto-pin migration

**Decision**: At first boot after C-FMC-6 deploys, pin every
auto-tracking campaign to the foundry-module's currently-installed
version. Subsequent installs go through the AutoPinHook flow.

**Idempotency**: stores the version it pinned to under
`foundry_vtt.autopin_migration_completed_for_version` in the
existing site_settings table. Subsequent boots read that key and
skip. Operator can re-trigger the migration by clearing the key
(intentional escape hatch).

**No-op cases**: no foundry-module package registered yet, or
package has no installed version. Returns nil without setting the
flag, so the next boot retries once the operator finishes setting
things up.

## HTMX-swap-safe pattern requirements

Logged here as a load-bearing reference. PRs #302, #303, C-FMC-5c,
and C-FMC-6 all hit production bugs from violating these. Every
interactive element in an HTMX-swappable fragment in this plugin
follows these rules:

1. **Inline `onclick="(function(){...})()"` IIFEs** — NOT
   `document.addEventListener` / delegated listeners, NOT templ
   `script` helpers. A `<script>` tag inside an HTMX-swapped
   fragment may not execute before the user clicks; inline
   `onclick` with the entire JS body in the attribute value works
   the moment the HTML is in the DOM.

2. **NEVER use templ `script` helpers in HTMX-loaded fragments.**
   This was a load-bearing bad pattern earlier in the consolidation.
   Templ scripts emit `<script>function __templ_X(){...}</script>`
   alongside the button with an onclick that references
   `__templ_X(...)`. The script tag and onclick reference race;
   browsers do NOT reliably execute innerHTML-swapped `<script>`
   tags synchronously, so the onclick can fire before the function
   is defined, raising:

   ```
   Uncaught ReferenceError: __templ_X is not defined
   ```

   PRs #302/#303 hit this with delegated listeners. C-FMC-5c's
   admin buttons + C-FMC-6's owner Save Pin and Rotate Token hit
   it with templ scripts. C-FMC-8 fixed all of them by routing
   every click handler through Go-side string builders in
   `onclick_handlers.go` that return a `templ.ComponentScript`
   with empty `Function` (no `<script>` tag) and the IIFE in
   `Call` (HTML-injected directly into the onclick attribute).

3. **Inline literal Alpine `x-data="{ open: false }"`** — never
   `x-data="someExternalFunction()"`. External references can be
   undefined when Alpine processes the directive after HTMX swap.

4. **`<details>` + `<summary>` for collapsible sections** when JS
   isn't needed for the toggle behavior. Browser-native, zero race
   window. Combine with `hx-trigger="toggle once from:closest details"`
   for lazy-load on expand.

5. **Defense-in-depth `style="display:none"`** alongside `x-show=open`
   for modal-like reveals. Prevents the brief flash if x-cloak
   evaluation runs late.

**Contract enforcement**: `onclick_handlers_test.go` asserts that
every handler's `Call` body (a) starts with `(function(`, (b)
contains no literal `"` (would close the `onclick="..."`
attribute), (c) contains no `__templ_` substring. Regressions to
the templ-script pattern fail these tests loudly.

## Database tables

### `foundry_vtt_campaign_tokens` (renamed from `foundry_module_campaign_tokens` in C-FMC-5c migration 001)

```
campaign_id   CHAR(36)  NOT NULL  PRIMARY KEY
token_version INT       NOT NULL  DEFAULT 1
rotated_at    TIMESTAMP NOT NULL  DEFAULT CURRENT_TIMESTAMP
FOREIGN KEY (campaign_id) REFERENCES campaigns(id) ON DELETE CASCADE
```

`token_version` is the rotation counter. Sign payload =
`HMAC(secret, "foundry-vtt:{campaign_id}:{token_version}")`. Rotate =
increment counter, every previously-signed URL becomes invalid on
next manifest request.

Table was originally created by `foundry_modules/migrations/001_*.up.sql`
(deleted in C-FMC-5c). The rename + foundry_modules deletion happens
atomically in `migrations/001_consolidate_foundry_modules.up.sql`.
Existing token rows are preserved by `RENAME TABLE`.

## Extension point: PostInstallHook

`foundry_vtt.NewPostInstallHook()` returns a struct satisfying
`packages.PostInstallHook` (added in C-FMC-5a). Registered at boot
via `packages.RegisterPostInstallHook(pkgService, hook)`.

Fires after every `PackageTypeFoundryModule` install:

1. Loads `chronicle-package.json` from the install dir (or applies
   defaults).
2. Reads `module.json` at the descriptor-declared `moduleJsonPath`.
3. Overwrites the `version` field with the installed version
   string.
4. Writes the file back.

Errors fail the install loudly (per C-FMC-5a contract). The
packages plugin's `InstallVersion` cleans up `destDir` on hook
failure, so a retry installs from scratch.

## Extension point: chronicle-package.json descriptor

The schema is owned on the Foundry side
(FM-PKG-DESCRIPTOR — `keyxmakerx/Chronicle-Foundry-Module#35`).
Chronicle's adoption is described here.

### Schema v1 (canonical)

```jsonc
{
  "schemaVersion": 1,
  "package": {
    "id": "chronicle-sync",
    "kind": "foundry-module",
    "moduleJsonPath": "module.json"
  },
  "serving": {
    "rewriteFields": ["manifest", "download"],
    "manifestEndpoint": "/api/v1/campaigns/{campaign_id}/foundry-vtt/module.json?token={token}",
    "downloadEndpoint": "/api/v1/campaigns/{campaign_id}/foundry-vtt/module.zip?token={token}",
    "perCampaignSignedToken": true,
    "zipContentRoot": ""
  }
}
```

### Fallback semantics

When no `chronicle-package.json` exists in the install dir,
`defaultDescriptor()` applies. The defaults match the canonical
schema above — equivalent to what the Foundry module's published
descriptor declares.

This means C-FMC-5b can deploy BEFORE FM-PKG-DESCRIPTOR ships —
old zips just use the fallback, new zips drive behavior from the
descriptor.

### Schema versioning

`schemaVersion` is the load-bearing field. The hook recognizes
only `currentSchemaVersion` (1 today). Unknown major versions
fail the install loudly. Future v2+ requires coordinated Foundry +
Chronicle releases.

Within v1: missing optional fields fall back to defaults via
`applyDefaults()`. Modules can ship minimal descriptors (just
`schemaVersion` + `package.id`) and inherit everything else.

## Error handling contract

Every public-facing error follows this format:

```
<what failed>: <technical detail>. <likely cause>. <actionable next step>.
```

See `errors.go` for all pre-defined constructors. Examples:

<!-- foundry-vtt-error-catalog-start -->
| Constructor | Category | HTTP Status | Format-compliant Message |
|---|---|---|---|
| `ErrInvalidToken` | auth | 403 | Wrong HMAC or rotated → operator rotates URL via campaign settings. |
| `ErrNoPackageRegistered` | config | 503 | No foundry-module package → admin adds it via `/admin/packages`. |
| `ErrPinnedVersionNotInstalled` | config | 503 | Pin to missing version → admin installs OR owner changes pin. |
| `ErrNoVersionAvailable` | config | 503 | Package exists, no version installed → admin clicks Install. |
| `ErrTokenNotInitialized` | config | 503 | First-time campaign hit before owner visited settings → owner opens VTT Setup Guides. |
| `ErrDescriptorInvalid` | validation | 422 | Bad descriptor in zip → fix upstream OR remove descriptor for defaults. |
| `ErrModuleJSONMissing` | internal | 500 | Zip malformed → admin retries OR upstream fix. |
| `ErrCampaignNotFound` | not_found | 404 | URL refers to deleted/wrong-instance campaign → owner mints a fresh URL. |
| `ErrInternal` | internal | 500 | Server-side fault; code is dynamic. Wildcard constructor; logged with the wrapped cause. |
<!-- foundry-vtt-error-catalog-end -->

JSON shape returned to public endpoints (Foundry's FM-CSU-DIAG
consumes this):

```json
{
  "error": "<code>",
  "message": "<format-compliant message>",
  "category": "<config|auth|not_found|validation|internal>"
}
```

### Adding a new error code

The catalog above is mechanically checked against `errors.go` by
the C-FMC-DRIFT-GUARD tests in `errors_test.go`. Three coupled
files must stay in sync; the tests fail loudly with an actionable
message if any pair drifts. Workflow:

1. **Add the constructor to `errors.go`** following the existing
   four-clause Message format. Pattern:

   ```go
   func ErrSomething(...) *Error {
       return &Error{
           Category: ErrCategoryConfig, // or ErrCategoryAuth, etc.
           Code:     "some_unique_snake_case_code",
           Message: "<what failed>: <technical detail>. " +
               "<likely cause>. <actionable next step>.",
       }
   }
   ```

2. **Add a row to the catalog table** above (inside the
   `<!-- foundry-vtt-error-catalog-start -->` /
   `<!-- foundry-vtt-error-catalog-end -->` markers) with the
   constructor name, category, HTTP status, and a one-line
   message summary. The HTTP status is derived from the category;
   the test enforces the mapping.

3. **Run `make foundry-error-catalog`** from the repo root to
   regenerate `error-catalog.json`. Commit the regenerated file.

4. **Run `go test ./internal/plugins/foundry_vtt/...`** —
   `TestErrorCatalog_*` will pass when all three files agree;
   they fail with a precise diff message until then.

If the Foundry module is consuming this code (it always is —
every code flows through the manifest endpoint), it requires
zero Foundry deploy: Foundry's `update-info.mjs` reads
`category` and `message` from Chronicle's response directly per
FM-CSU-DIAG-FIX (Foundry-side PR #40). Adding new codes is a
Chronicle-only operation.

### Drift guard scope (established by C-FMC-5c / #306; future PRs may extend)

The drift guard mechanically aligns three sources:

- `errors.go` — typed constructors (source of truth for Code +
  Category)
- `.ai.md` — human-authored catalog table (consumer-facing docs)
- `error-catalog.json` — machine-readable artifact (consumed by
  Foundry-side docs + potentially Foundry-side CI)

The shared parser lives in `errors_catalog.go` and is used by both
the test path (`errors_test.go`) and the regeneration path
(`cmd/foundry-error-catalog/main.go`).

The canonical catalog also publishes at
`/internal/plugins/foundry_vtt/error-catalog.json`. Foundry's
repo references this artifact in its own docs (`.ai.md` and
`API-CONTRACT.md`). Future hardening could fetch this URL in
Foundry-side CI and assert no drift — out of scope for C-FMC-
DRIFT-GUARD; tracked as FM-DRIFT-GUARD for a later dispatch.

### Standing pre-merge rule (C-FMC-DRIFT-GUARD addition)

**Catalog drift trace** — every PR touching `errors.go` (adding,
removing, or modifying constructors) must also update the
`.ai.md` catalog table inside the markers AND regenerate
`error-catalog.json` via `make foundry-error-catalog`. The drift
tests enforce this mechanically; reviewers don't need to
remember to check three files manually.

This sits alongside the C-FMC-8 codified rules:

- Fragment consumer trace (every PR creating an hx-get'd
  fragment lists consumers)
- On-disk artifact trace (every PR changing how a URL gets
  served verifies the on-disk artifact carries the new URL)
- HTMX click-handler trace (every PR adding interactive
  elements to an HTMX fragment uses inline IIFE)

Together: four mechanical safeguards against the recurring
"forgotten consumer / wrong contract / silent regression" class
of bug that surfaced across C-FMC-5 through C-FMC-9.
