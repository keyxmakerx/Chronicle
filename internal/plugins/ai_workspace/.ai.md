# ai_workspace plugin

## Purpose

Chronicle's AI co-pilot surface. Owns:

- **AI Export** — the markdown-for-AI generator (relocated from
  `internal/aiexport/` in C-AI-WORKSPACE-V1-B; functionally identical
  to PR #350)
- **Prompt builder** — Phase 3 (`C-AI-WORKSPACE-V1-C`)
- **AI Import** — Phases 4-5 (`C-AI-WORKSPACE-V1-D` + `-V1-E`)

First plugin added under the post-NW-2.2 plugin-isolation rules.
Doubles as a showcase of the pattern: own settings tab via
`campaigns.RegisterSettingsTab`, own routes via
`RegisterOwnerRoutes`, owns no shared state beyond the renderer
service.

**Status: V1 ARC SHIPPED** — V1-A through V1-F (PRs #351-#356)
closed the end-to-end loop: Generate prompt → paste into AI → paste
result back → Review → Submit → entities created with sanitized
content + new categories. Owner-only; production-ready.

## Tier

**Plugin** under `internal/plugins/ai_workspace/`. Cross-cutting in
nature — depends on every other plugin's lister interface via the
nested `aiexport/` sub-package, but exports nothing visible to those
plugins. The campaigns plugin is the only consumer (via the
SettingsTab registry and the parent route group).

## Wiring

`internal/app/routes.go` does the construction at startup:

```go
aiWorkspaceRenderer := aiexport.NewService(entityService, noteSvc, ...)
aiWorkspaceHandler := ai_workspace.NewHandler(aiWorkspaceRenderer)
aiWorkspaceHandler.SetAuditLogger(&aiWorkspaceAuditAdapter{svc: auditService})

// Contribute the settings tab (slot 55 — between AI Export's old
// slot 50 and Activity's slot 60; pinned in V1-A's
// TestRegisterSettingsTab_MergesAndSorts).
campaignHandler.RegisterSettingsTab(aiWorkspaceHandler.SettingsTabFactory())

// Mount routes on a /campaigns/:id group with auth + membership.
aiWorkspaceCampaignAuthed := e.Group("/campaigns/:id",
    auth.RequireAuth(authService),
    campaigns.RequireCampaignAccess(campaignService),
)
ai_workspace.RegisterOwnerRoutes(aiWorkspaceCampaignAuthed, aiWorkspaceHandler,
    campaigns.RequireRole(campaigns.RoleOwner))
```

## Settings tab integration

`SettingsTabFactory` returns a `func(*campaigns.CampaignContext) campaigns.SettingsTab` factory. Campaigns invokes it per-request inside `visibleSettingsTabs`, so the tab's `Content` closure can bind `cc.Campaign.ID` into the form's `hx-get` URL. Slot 55 (between AI Export's retired slot 50 and Activity's slot 60); the V1-A test `TestRegisterSettingsTab_MergesAndSorts` already pinned this slot expecting V1-B to fill it.

## Audit

Each successful `GenerateAIExport` invocation fires `campaign.ai_export.generated` with privacy mode + category count + GM-notes opt-in flag + markdown byte count. No markdown body in the audit (audit log is events, not bulk content). Audit hook is optional via `SetAuditLogger`; nil disables emission for test fixtures.

## V1.5 verb-set extension (C-AI-WORKSPACE-V1-G; Wave 3 Phase 1)

V1.5 extends the V1 verb set with explicit action verbs in front-
matter — closing the gap where AI could create entities but couldn't
explicitly express "edit this one" or "delete this one" via the import
flow. Per Q-V2-AI-1 resolution (locked 2026-05-28) + the Phase 0 audit
at `cordinator/reports/chronicle/2026-05-28-c-ai-workspace-destructive-
scoping.md`.

### Front-matter `action:` field

```yaml
---
type: character
action: create    # OR update OR delete; default "create"
name: Eldoria the Wise
visibility: private
tags: [scholar, ally]
---

Body markdown for the entity's wiki page.
```

| Action | Required FM fields | Body | Behavior |
|---|---|---|---|
| `create` (default) | `name` (or H1) + `type` (or bulk default) | Required | V1 behavior — create new entity; conflict-mode dropdown on slug collision |
| `update` | `name` (or H1) | Required | Load existing entity by slug → replace metadata + body (same semantic as V1's ConflictMode=overwrite) |
| `delete` | `name` (or H1) | Ignored (warning emitted if present) | Load existing entity by slug → call `EntityService.Delete`; soft vs hard delegated to the entities plugin |

Omitted `action:` field defaults to `create` — preserves V1-era AI prompts
without modification. Invalid action value yields `StatusParseError`.

### Conflict-mode vocabulary

V1.5 renames `Overwrite` → `Update` in the conflict-mode dropdown for
vocabulary consistency with the new action verb. The commit handler
accepts `conflict=overwrite` as a backward-compat alias for one release;
removed in V2.

| Form value | UI label | Behavior |
|---|---|---|
| `skip` (unchanged) | Skip | Unchanged |
| `rename` (default; unchanged) | Rename (suffix) | Unchanged — `(Imported)` suffix |
| `update` (renamed from `overwrite`) | Update existing | Unchanged — same semantic as V1 Overwrite |

### Delete confirmation UI

Per the audit §5.2 + dispatch §D recommendation: each `action: delete`
row gets an inline "I understand X will be permanently deleted"
confirmation checkbox with `required` attribute. Browser-native form
validity blocks Submit until every Delete row's confirmation is checked.
The committer re-checks `DeleteConfirmed` belt-and-suspenders against
any client-side bypass.

### Audit-log taxonomy (counts-only; V1-E discipline preserved)

`campaign.ai_import.committed` payload:

| V1 key | V1.5 key |
|---|---|
| `created` | `created` (unchanged) |
| `renamed` | `renamed` (unchanged) |
| `overwrote` | **`updated`** (renamed) |
| (new) | **`deleted`** |
| `skipped` | `skipped` (unchanged) |
| `failed` | `failed` (unchanged) |
| `new_categories_created` | `new_categories_created` (unchanged) |
| `new_categories_failed` | `new_categories_failed` (unchanged) |

Pre-V1.5 audit-log entries with `overwrote: N` continue to exist
verbatim; new entries use `updated: N`. The operator audit UI renders
JSON keys verbatim — no parser brittleness.

### Sanitize AST pin

`committer_sanitize_test.go::TestCommitter_EntityMutationsFunnelThroughMarkdownToHTML`
extends cleanly:

- `commitUpdate` (renamed from `overwriteExisting`) is exempted with
  the same reason — caller funnels through `MarkdownToHTML` before
  invoking
- `commitUpdateExplicit` (new function calling the existing
  `commitUpdate` exempt path) MUST funnel through `MarkdownToHTML` —
  the pin fires correctly because the AST pin only checks for the
  funnel within the function body that calls `creator.Update`
- `commitDelete` is naturally exempt — `creator.Delete` is NOT in the
  pin's `mutators` map; no body to sanitize
