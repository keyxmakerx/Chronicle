# Bestiary Plugin

## Purpose

The Community Bestiary enables instance-wide sharing of homebrew creature statblocks.
Users publish creatures from their campaigns, and all authenticated users on the
instance can browse, search, rate, favorite, and import them into their own campaigns.
Instance admins can moderate flagged content.

**No UI in this plugin.** All handlers return JSON. The UI is provided by system
package widgets (e.g., the Draw Steel monster builder widget) that call these endpoints.

## Multi-System Support

The bestiary is **not tied to Draw Steel**. Every publication has a `system_id` field
(VARCHAR 100, defaults to `"drawsteel"` if omitted). The search API accepts
`system_id` as a filter parameter. Widgets should always pass `system_id` in
browse/search calls to show only creatures for their game system.

## Key Architectural Decisions

- **Instance-scoped, not campaign-scoped.** Routes live at `/bestiary/*`, not under
  `/campaigns/:id/`. Publications are visible to all authenticated users.
- **Statblock JSON stored as-is.** The `statblock_json` column stores the complete
  creature definition as opaque JSON. The backend validates it's valid JSON under
  100KB and extracts `level`, `organization`, and `role` for indexed filtering, but
  does NOT enforce game-system-specific schemas. Schema validation is the widget's
  responsibility.
- **Slug-based URLs.** Publications are accessed by slug (`/bestiary/ashen-wyrm`)
  for shareability.
- **Transactional writes.** Ratings, favorites, and flags use `SELECT ... FOR UPDATE`
  row locking inside database transactions to prevent race conditions from concurrent
  requests. Count adjustments (rating_sum, rating_count, favorites, flagged_count)
  are always inside the same transaction as the row modification.
- **Per-user flag deduplication.** The `bestiary_flags` table (migration 002) has a
  composite PK `(user_id, publication_id)` that prevents one user from flagging the
  same publication multiple times. Auto-hide triggers at 3+ unique user flags.
- **Per-user rate limiting.** In-memory token bucket middleware keyed by user ID,
  applied at the route level. Limits per design doc (10/hr publish, 30/hr rate, etc.).

## Database Tables

| Table | Purpose |
|-------|---------|
| `bestiary_publications` | Core table — creature statblocks with denormalized level/org/role for filtering. FULLTEXT index on name+description. |
| `bestiary_ratings` | 1-5 star ratings with optional review text. UNIQUE(user_id, publication_id). |
| `bestiary_favorites` | Bookmark table. Composite PK (user_id, publication_id). |
| `bestiary_imports` | Tracks which publications were imported into which campaigns. UNIQUE(publication_id, campaign_id). |
| `bestiary_moderation_log` | Audit trail for admin moderation actions. |
| `bestiary_flags` | Per-user flag tracking. Composite PK prevents duplicate flags. |

## Cross-Plugin Dependencies

All adapters are defined in `internal/app/routes.go`:

| Interface (in bestiary) | Adapter | Wraps |
|------------------------|---------|-------|
| `UserFetcher` | `bestiaryUserFetcherAdapter` | `auth.AuthService.GetUser` — creator display names/avatars |
| `EntityCreator` | `bestiaryEntityCreatorAdapter` | `entities.EntityService.Create` — import creates campaign entities |
| `CampaignRoleChecker` | `bestiaryCampaignRoleAdapter` | `campaigns.CampaignService.GetMember` — import verifies Scribe+ role |

**Note:** The `EntityCreator` adapter currently passes statblock as `fields_data["statblock_json"]`.
The Draw Steel system package may need to define how this maps to its creature preset's fields
(abilities_json, villain_actions_json, etc.).

## Widget Integration Guide

The Draw Steel monster builder widget (or any system's creature widget) should:

1. **Publish:** `POST /bestiary` with `{ system_id, name, statblock_json, visibility, tags?, description? }`
2. **Browse:** `GET /bestiary/search?system_id=drawsteel&q=...&level_min=...&sort=newest`
3. **View:** `GET /bestiary/:slug` (full pub) or `GET /bestiary/:slug/statblock` (JSON only)
4. **Rate:** `POST /bestiary/:id/rate` with `{ rating: 1-5, review_text?: "..." }`
5. **Favorite:** `POST /bestiary/:id/favorite` (toggles on/off)
6. **Import:** `POST /bestiary/:id/import/:campaignId` (creates entity in campaign)

All responses are JSON. Errors follow Chronicle's standard format:
`{ "type": "not_found", "message": "publication not found" }`.

## What's NOT in This Plugin

- **No UI/templates** — JSON API only, no `.templ` files
- **No statblock schema enforcement** — beyond basic JSON + size validation
- **No cross-instance federation** — planned for Phase 2 of the design doc, far future
- **No unit tests yet** — service/repo tests with mocked boundaries are a TODO
