# Sync API Plugin

## Purpose

Provides a versioned REST API (v1) for external tool integration with Chronicle.
External clients such as Foundry VTT modules and custom scripts use these
endpoints to read and write campaign data programmatically. Also provides a
bidirectional entity sync protocol, API key management, request logging,
security event tracking, and an admin monitoring dashboard.

## Tier

**Plugin** -- full feature app with handlers, service, repository, middleware,
and admin templates.

## How It Works

### Authentication

API keys use a prefix-lookup + bcrypt-verify scheme:

1. Client sends `Authorization: Bearer chron_<hex>` header.
2. Middleware extracts the 8-char prefix, looks up the key row by prefix.
3. Full key is verified against the stored bcrypt hash.
4. Key must be active and not expired.

Keys are scoped to a single campaign and carry permissions (`read`, `write`, `sync`).

### Device Fingerprint Binding

Clients may send `X-Device-Fingerprint` header. On first use the fingerprint is
bound to the key. Subsequent requests from a different fingerprint are rejected
with a security event logged. This prevents key sharing across devices.

### Rate Limiting

Fixed-window counter per API key per minute. The limit is configurable per key
(default 60, max 1000). Response headers `X-RateLimit-Limit` and
`X-RateLimit-Remaining` are set on every response. Exceeding the limit returns
429 with `Retry-After: 60`.

### Addon Gating

Calendar and maps API endpoints are gated behind `RequireAddonAPI` middleware,
which checks whether the addon is enabled for the campaign. Returns 404 when
disabled, 503 when the check fails (fail-closed).

### Middleware Stack

API v1 has two parallel groups at the `/api/v1` prefix (per operator decision D-C3.1 — sub-group skip, PR #344):

- **`v1` (JSON group)**: `RequireAuthOrAPIKey` → `RateLimit` → `RequireJSONContentType` → (per-route) `RequireCampaignMatch` → `RequirePermission` → optionally `RequireAddonAPI`. `RequireJSONContentType` rejects state-changing requests (POST/PUT/PATCH) without `Content-Type: application/json` with 415; safe methods (GET/HEAD/DELETE/OPTIONS) pass through.
- **`v1Multipart` (multipart sub-group)**: `RequireAuthOrAPIKey` → `RateLimit` → (per-route) `RequireCampaignMatch` → `RequirePermission`. Skips `RequireJSONContentType` so `multipart/form-data` is accepted. Inhabitant today: only `POST /api/v1/campaigns/:id/media` (`UploadMedia`). Any future multipart endpoint under `/api/v1/*` mounts here, not on `v1`.

### Egress HTML sanitization

Per C-SEC-6-AMENDED (PR #345), the 6 `/api/v1/*` GET handlers that emit user HTML — `GetEntity`, `ListEntities`, `GetNote`, `ListNotes`, `GetEvent`, `ListEvents` — invoke an `egress_sanitize.go` helper before `c.JSON(...)` to re-sanitize the HTML fields on the response copy. `sanitize.HTMLPtr` is the nullable companion to `sanitize.HTML`; the helpers operate on fresh DB-scanned models so the canonical row is untouched.

This is the **defense-in-depth EGRESS layer**. INGRESS sanitization on the write path (in each source plugin's `service.go`) is the primary defense; the egress layer covers historical rows or tooling-bug-inserted rows that slipped past ingress. Scope is `/api/v1/*` Foundry-bound surfaces ONLY — the backup/restore round-trip (`internal/app/export_adapters.go`, `campaigns/export_handler.go`, `internal/plugins/restore/`) is the §0.5 D4=(c) lossless carve-out and MUST NOT be re-sanitized. The structural pin `TestEgressSanitize_HandlersInvokeHelpers` enforces the wiring at every named handler.

## Sync Protocol

The `POST /sync` endpoint supports bidirectional sync in a single request:

- **Pull**: set `since` (RFC3339 timestamp) to receive entities modified after
  that time. Paginated internally (max 10 pages of 100). Returns `has_more` flag.
- **Push**: set `changes` array with `{action, entity_id, ...}` objects.
  Actions: `create`, `update`, `delete`. Max 2000 changes per request.
- **Response**: `server_time` for use as the next `since` value, `entities`
  array of pulled data, and `results` array with per-change status.

## Notable Design Decisions

- Keys use bcrypt hashing with prefix-based lookup to avoid full-table scans.
- Rate limiter is in-memory (not Redis) for simplicity; acceptable for
  single-instance deployments.
- Request logging and security events are fire-and-forget goroutines to avoid
  blocking API responses.
- IDOR protection on every endpoint: entity/event/media ownership verified
  against the key's campaign before any operation.
- WebSocket authentication reuses the same key validation path via
  `AuthenticateKeyForWS`, returning owner-level role for API key holders.
- Media API responses include HMAC-signed URLs when a signer is configured,
  with 1-hour expiry for direct file access.
- **Calendar date beacon (C-SYNC-DATE-BEACON, 2026-07-17)** — lights up the
  calendar sky strip's dormant sync-chip DRIFT state (chronicle#545 flagged
  this gap: the chip's drift logic was built + unit-tested, but nothing
  recorded what date Foundry last SAW, and `Calendar.UpdatedAt` can't
  distinguish a Foundry push from a Chronicle-UI edit). `GetCurrentDate`
  (`calendar_api_handler.go`) now records `{campaign_id, last_served_year/
  month/day, last_served_at}` on every read — deliberately named "served",
  never "applied"/"synced". **Storage:** new table
  `sync_calendar_date_beacons` (migration `005_calendar_date_beacon`, PK
  `campaign_id`) rather than a column on an existing row — `api_keys` is
  per-KEY (a campaign can have several) and `sync_mappings` is per-synced-
  OBJECT; neither is per-campaign grain. **Throttle:** skip the write if
  the existing beacon is <60s old AND the date is unchanged
  (`syncAPIService.RecordCalendarDateBeacon`, `calendarDateBeaconThrottle`);
  a changed date always writes immediately even if the last write was
  seconds ago — freshness for the chip matters more than throttle
  discipline once the date has actually moved. The write itself is
  fire-and-forget on a background context
  (`recordCalendarDateBeaconIfModule`, same convention as
  `UpdateKeyLastUsed`/`LogRequest` in middleware.go) so the throttle SELECT
  + conditional UPSERT never blocks the GET response. **CRITICAL auth
  gate:** `GET /calendar/date` is dual-auth (`RequireAuthOrAPIKey` — a
  session cookie OR a Bearer key both work). Only a REAL Bearer key
  (`GetAPIKey(c).ID != synthKeySessionID`) may beacon; a member browsing
  Chronicle's own calendar over the session-cookie path on this same
  endpoint must never beacon (a Chronicle-UI page view isn't "the module
  reading a date"). Pinned by 3 handler tests
  (`calendar_date_beacon_handler_test.go`). **Read side:** a NEW
  member-read endpoint `GET /campaigns/:id/calendar-sync-beacon`
  (`Handler.GetCalendarSyncBeacon`, `routes.go` `RegisterCampaignRoutes` —
  no `RequireRole`, any campaign member) rather than folding into
  `foundry_vtt`'s `/foundry-presence` payload — the two plugins have zero
  existing Go coupling in either direction, and adding one just for this
  would cost more than a second small endpoint (T-B2). `computeSyncChipState`
  in `calendar_v2_shell.js` (calendar plugin) now fetches both endpoints
  client-side (plain HTTP, no cross-plugin import) and feeds its
  previously-dormant `fmConfirmedDate`/`chronicleCurrentDate` params —
  drift is reachable in production now, not just by hand-fed unit tests.
