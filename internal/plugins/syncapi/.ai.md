# Sync API Plugin

## Purpose

Provides a versioned REST API (v1) for external tool integration with Chronicle.
External clients such as Foundry VTT modules and custom scripts use these
endpoints to read and write campaign data programmatically. Also provides a
bidirectional entity sync protocol, API key management, request logging,
security event tracking, and an admin monitoring dashboard.

## Tier

**Plugin** -- full feature app with handlers, service, repository, middleware,
and admin templates.

## How It Works

### Authentication

API keys use a prefix-lookup + bcrypt-verify scheme:

1. Client sends `Authorization: Bearer chron_<hex>` header.
2. Middleware extracts the 8-char prefix, looks up the key row by prefix.
3. Full key is verified against the stored bcrypt hash.
4. Key must be active and not expired.

Keys are scoped to a single campaign and carry permissions (`read`, `write`, `sync`).

### Device Fingerprint Binding

Clients may send `X-Device-Fingerprint` header. On first use the fingerprint is
bound to the key. Subsequent requests from a different fingerprint are rejected
with a security event logged. This prevents key sharing across devices.

### Rate Limiting

Fixed-window counter per API key per minute. The limit is configurable per key
(default 60, max 1000). Response headers `X-RateLimit-Limit` and
`X-RateLimit-Remaining` are set on every response. Exceeding the limit returns
429 with `Retry-After: 60`.

### Addon Gating

Calendar and maps API endpoints are gated behind `RequireAddonAPI` middleware,
which checks whether the addon is enabled for the campaign. Returns 404 when
disabled, 503 when the check fails (fail-closed).

### Middleware Stack

API v1 requests pass through: `RequireAPIKey` (auth + IP blocklist + request
logging) -> `RateLimit` -> `RequireCampaignMatch` -> `RequirePermission` ->
optionally `RequireAddonAPI`.

## Sync Protocol

The `POST /sync` endpoint supports bidirectional sync in a single request:

- **Pull**: set `since` (RFC3339 timestamp) to receive entities modified after
  that time. Paginated internally (max 10 pages of 100). Returns `has_more` flag.
- **Push**: set `changes` array with `{action, entity_id, ...}` objects.
  Actions: `create`, `update`, `delete`. Max 2000 changes per request.
- **Response**: `server_time` for use as the next `since` value, `entities`
  array of pulled data, and `results` array with per-change status.

(with resolve), IP blocklist management (block/unblock), and key toggle/revoke.

## Notable Design Decisions

- Keys use bcrypt hashing with prefix-based lookup to avoid full-table scans.
- Rate limiter is in-memory (not Redis) for simplicity; acceptable for
  single-instance deployments.
- Request logging and security events are fire-and-forget goroutines to avoid
  blocking API responses.
- IDOR protection on every endpoint: entity/event/media ownership verified
  against the key's campaign before any operation.
- WebSocket authentication reuses the same key validation path via
  `AuthenticateKeyForWS`, returning owner-level role for API key holders.
- Media API responses include HMAC-signed URLs when a signer is configured,
  with 1-hour expiry for direct file access.

## Dashboard Integration

- `sync_status` dashboard block type for campaign dashboard editor
- `GET /campaigns/:id/sync-status` — HTMX fragment showing active API key count,
  24h request stats, and per-key status with last-used timestamps (Owner only)
- Lazy-loaded via `hx-trigger="intersect once"` in the dashboard block
