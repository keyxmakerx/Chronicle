# Packages Plugin

The packages plugin manages external package repositories for Chronicle. It pulls
content packages (game systems and other extensions) from GitHub repos, providing
version management, auto-updates, and an admin UI.

After C-FMC-5c, this plugin is **fully type-agnostic**: it knows nothing about
the specifics of any particular package type. The `PackageType` enum (currently
`system` and `foundry-module`) is just a label admins attach to a package;
type-specific behavior is owned by sub-plugins that register via the
`PostInstallHook` extension point. The first such sub-plugin is `foundry_vtt`,
which owns every Foundry-specific behavior (manifest rewriting, per-campaign
signed URLs, descriptor reading). Future TTRPG VTT integrations (Roll20,
Owlbear) follow the same pattern.

## Architectural decision log

| Decision | When | Why |
|---|---|---|
| Foundry coupling removed from this plugin | C-FMC-5c | Operator's modularity directive: "we don't want foundry stuff inside chronicle natively, because then that just makes a redundant mess." Foundry-specific behavior now lives entirely in the `foundry_vtt` sub-plugin; this plugin is generic and serves any package type via `PostInstallHook`. |
| `PackageType` enum kept | C-FMC-5c | The string label is still useful — admins need to declare what kind of package they're adding (system vs. foundry-module) so the right sub-plugin's hook fires. Removing the enum would require sub-plugins to inspect the zip's contents to identify themselves, which is more brittle. |
| `PostInstallHook` extension point | C-FMC-5a | Cross-plugin extension primitive: sub-plugins attach type-specific install-time behavior without packages having to know about them. Hooks fail loudly on errors (operator-actionable failures vs. silent stale state). |
| Multi-version preservation by default | C-FMC-5b (verified C-FMC-5c) | `InstallVersion` writes to a per-version directory and never deletes old ones. Lets foundry_vtt serve campaigns pinned to historical versions even after the admin has installed a newer one. `PruneStaleVersions` (shipped PR #513: `service.go` + `prune.go` + `handler.go` dry-run/execute + wizard) lets operators reclaim old on-disk version dirs, keeping the N newest. |

## Key Concepts

- **PackageType**: `system` (game content packs) or `foundry-module` (Foundry VTT sync module)
- **UpdatePolicy**: `off`, `nightly`, `weekly`, `on_release` — controls auto-update frequency
- **Version pinning**: Locks a package to a specific version, preventing auto-updates
- **GitHub integration**: Uses GitHub Releases API; supports GITHUB_TOKEN for higher rate limits

## Database Tables

- `packages` — registered package repositories with installed state
- `package_versions` — discovered versions from GitHub releases

## Admin Routes

All under `/admin/packages`, requiring site admin authentication:
- `GET /admin/packages` — package management page
- `POST /admin/packages` — add new package repo
- `DELETE /admin/packages/:id` — remove package
- `GET /admin/packages/:id/versions` — list versions (HTMX fragment)
- `PUT /admin/packages/:id/version` — install version
- `PUT /admin/packages/:id/pin` — pin version
- `DELETE /admin/packages/:id/pin` — unpin
- `PUT /admin/packages/:id/auto-update` — change policy
- `POST /admin/packages/:id/check` — manual update check
- `GET /admin/packages/:id/usage` — campaign usage (HTMX fragment)

## Extension points

### `PostInstallHook` (added C-FMC-5a)

Sub-plugins attach type-specific behavior to the install pipeline via
`packages.PostInstallHook`:

```go
type PostInstallHook interface {
    PackageType() PackageType
    AfterInstall(ctx context.Context, pkg *Package, version, destDir string) error
}
```

Registered at boot via `packages.RegisterPostInstallHook(svc, hook)`.
Hooks fire after the zip is extracted, validated, and the DB row
updated. The packages plugin dispatches by `PackageType` — a hook
registered for `PackageTypeFoundryModule` only fires for foundry-module
installs.

Errors fail the install loudly + cleanup `destDir` so a retry installs
cleanly. No silent warn-and-continue.

First consumer: `foundry_vtt` (C-FMC-5b) registers a hook that reads
`chronicle-package.json` and rewrites `module.json` version. This is
the mechanism that fixes the operator's "Foundry shows v0.1.0 after
installing v0.2.0" bug end-to-end.

### `InstallDirForVersion` (added C-FMC-5b)

`PackageService.InstallDirForVersion(pkgType, slug, version) string`
returns the on-disk path for a specific historical version of a
package, regardless of which version is marked "installed" in the
DB. Used by sub-plugins to serve a campaign-pinned version that may
not be the package's currently-active install.

Path is computed deterministically; no filesystem check is
performed. Callers stat the returned path to confirm the version
is actually installed on disk.
