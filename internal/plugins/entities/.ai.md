## Purpose

The heart of Chronicle. Manages the universal entity system where every
worldbuilding object (character, location, item, etc.) is an entity with
a configurable type. Handles entity CRUD, configurable entity types with
dynamic fields, entity search, nesting, and the entity profile page.

## Domain Concepts

- **Entity:** A worldbuilding object with a name, type, rich text entry,
  and configurable fields. Everything is an entity.
- **Entity Type:** A configurable template defining what fields an entity
  has. Per-campaign. Defaults ship pre-configured (Character, Location, etc.)
  but can be customized by the GM.
- **Fields Definition:** JSON array on entity_types defining field key, label,
  type, options, and display section.
- **Fields Data:** JSON object on an entity storing that entity's values for
  its type's fields.
- **Entry:** The rich text body of an entity (TipTap/ProseMirror JSON).
- **Player Notes:** Separate player-facing content (`player_notes` JSON +
  `player_notes_html` HTML). Distinct from the main entry (DM content). Maps to
  a second Foundry Journal page.
- **Privacy:** is_private flag makes an entity visible only to Scribes and Owners.
  Players see 404 for private entities. Owned end-to-end by the slide-in
  permissions card (`static/js/widgets/permissions.js`); the entity edit
  form and the inline metadata panel no longer carry an is_private input
  (dispatch C-PERMISSIONS-INLINE-COMPONENT, 2026-05-19). The card writes
  via PUT `/entities/:eid/permissions`. `UpdateEntityInput.IsPrivate` is
  `*bool` so the form/metadata handlers can pass `nil` and have the
  service preserve the entity's existing value — without this, removing
  the UI toggle silently flipped every entity to public on next save.
- **Optimistic Concurrency:** `expected_updated_at` on update requests triggers
  409 Conflict if the entity was modified since that timestamp.

## Sidebar Navigation System

The entities plugin manages the sidebar drill panel — the slide-over
category view with its entity tree, folder nodes, favorites, and search.

### Dual-Parent Model

Entities have two mutually exclusive parent references:
- `parent_id` (CHAR(36) FK→entities) — parent is another entity
- `parent_node_id` (CHAR(36) FK→sidebar_nodes) — parent is a folder node

When one is set, the other must be NULL. The service layer enforces this
in `ReorderEntity()`. The sidebar_tree.js tree builder resolves both.

### Sidebar Nodes (Pure Folders)

The `sidebar_nodes` table stores organizational folders that have NO entity
record. They appear only in the sidebar tree as non-navigable grouping
containers. CRUD via `/campaigns/:id/sidebar-nodes`.

### Favorites

The `entity_favorites` table stores per-user, per-campaign bookmarks.
Replaces the localStorage-based system for cross-device persistence.
API at `/campaigns/:id/entities/:eid/favorite` (toggle) and
`/campaigns/:id/favorites` (list).

### Tag Filtering

The search API accepts `?tags=slug1,slug2` for AND-logic tag filtering.
Uses a subquery with `HAVING COUNT(DISTINCT)` for correctness.

### Lazy Loading

Sidebar drill panel loads 50 entities per page. The template renders a
"Load more" sentinel when `total > len(results)`. JS IntersectionObserver
auto-triggers next-page fetches.

## Business Rules

- Entity names must be non-empty (max 200 chars)
- Slug auto-generated from name, deduplicated with -2/-3 suffix, unique per campaign
- Entity type determines which fields appear in the profile and edit form
- Dynamic fields parsed from form params (field_<key>) by handler
- Private entities (is_private=true) filtered at SQL level: Players don't see them
- Show handler returns 404 (not 403) for private entities to avoid revealing existence
- FULLTEXT search on entity name (BOOLEAN MODE), LIKE fallback for queries < 4 chars
- Deleting an entity cascades via FK (future: posts, tags, relations)
- Default entity types seeded on campaign creation via EntityTypeSeeder interface
- 6 default types: Character, Location, Organization, Item, Note, Event
- **Claimable (PC-CLAIM-2):** `entity_types.claimable BOOLEAN NULL` (migration 000029).
  NULL = unset → `isClaimableType` falls back to the legacy heuristic
  (preset_category "character" / `*-character` slug); TRUE/FALSE = the Owner's
  explicit choice and overrides the heuristic. Creating a player-character
  sub-type (`PresetCategory=="player_character"` or `slug=="player-character"`)
  requires the `player-character-claiming` addon enabled for the campaign —
  `CreateEntityType` gates it via the injected `AddonChecker` (`SetAddonChecker`)
  and defaults `claimable=true` when the addon is on.
- **Claiming UI (PC-CLAIM-3):** three surfaces, all gated through the handler's
  `isAddonEnabled(…, AddonPlayerCharacterClaiming)` (fails open) + `isClaimableType`.
  (1) `claim_banner.templ` (`claimBanner`) on the show page: "Claimed by &lt;name&gt;"
  when `OwnerUserID != nil` (Show handler resolves the id → `CampaignMember.DisplayName`
  via `memberLister`), else the unclaimed claim banner — which now requires addon-on
  AND claimable. (2) `category_owner_roster.templ` (`claimRosterPanel`): the Scribe+
  owner overview on a claimable category dashboard; the Index handler builds a
  `ClaimRoster` (model.go — full type list capped 100 + members + owner-name map) and
  threads it through `CategoryDashboardContent` **inside `#entity-list`** so the
  search/sort `outerHTML` swap is preserved; reassign/unclaim PUT `…/entities/:eid/owner`.
  (3) `EntityTypeCard` + the Add-Category form carry the "Players can claim" checkbox
  only when the addon is on (`claimingEnabled` threaded through `EntityTypesManagePage`
  → `EntityTypeListContent` → `EntityTypeCard`). Pure helpers in `claim_overview.go`
  (`ownerDisplayNames`/`resolveOwnerName`); tests in `claim_overview_test.go`.

## Player Character Claiming (PC-CLAIM)

A feature (Stages 1–3 merged) for linking player-owned entities (typically
characters) to campaign members, enabling GMs to manage ownership and players
to claim their own characters. This section documents the complete flow and
architecture across three stages.

### Overview & Motivation

Without character ownership, a GM cannot see who plays which character, and
campaigns lack a structured player-character binding. The feature allows:
- **Players** claim unclaimed character entities
- **GMs** see who owns what, reassign ownership, or unclaim entities
- **Audit trail** visible in the activity log (distinct `entity.claimed` and
  `entity.owner_changed` actions)
- **Stage 4** ✅ DELIVERED (FM #64) — auto-claim of Foundry actor ownership for seamless
  sync when a player-owned actor is imported

### Stage 1: Audit Visibility (merged PR #480)

Added `entity.claimed` and `entity.owner_changed` audit actions to the
activity log, making claims and ownership changes legible in the activity
feed. The claiming user's name and the character's real name are recorded,
yielding audit entries like "Alice claimed Tyne" instead of the prior
generic "entity updated" with an opaque user ID.

**Files:** `internal/plugins/audit/{model.go, activity.templ}`

**Service methods:** `ClaimEntity` + `AssignOwner` + `logAudit` / `logAuditWithDetails`

### Stage 2: Database & Logic Layer (merged PR #481)

Plumbed the database column and service-layer logic:

**Database (migration 000029):** `entity_types.claimable BOOLEAN NULL`
- NULL (default) = unset; fall back to legacy heuristic (preset_category
  "character" or slug `*-character`)
- TRUE/FALSE = the Owner's explicit override of the heuristic

**Service logic (`isClaimableType`):** checks `claimable` first; falls back
to heuristics if NULL. Honors the Owner's choice for fine-grained control.

**Addon gate:** Creating a "Player Character" sub-type
(preset_category="player_character" or slug="player-character") requires
the `player-character-claiming` addon to be enabled. `CreateEntityType`
gates the creation via an injected `AddonChecker` interface and defaults
`claimable=true` when the addon is on.

**Model updates:** `EntityType.Claimable *bool` field, scan path unified.

**Files:** `model.go`, `repository.go`, `service.go`, `db/migrations/000029_*.sql`

### Stage 3: UI Layer (merged PR #482)

Exposed claiming features across four surfaces, all gated on
`addon-enabled(player-character-claiming)` AND `isClaimableType(type)`:

1. **Claim banner on entity show page** (`claim_banner.templ`):
   - **Unclaimed:** "Claim this character" button (Player+ action)
   - **Claimed:** "Claimed by <DisplayName>" badge (read-only)
   - Show handler resolves `owner_user_id` → campaign member display name
     (or fallback "a player" for stale/unresolved owner)

2. **Owner roster on category dashboard** (`category_owner_roster.templ`):
   - Scribe+ only, appears on any claimable type's dashboard
   - Table: character name + current owner + reassign select + unclaim button
   - Roster capped at 100 entities/category (one page)
   - Both actions hit `PUT /campaigns/:id/entities/:eid/owner`

3. **Per-type claiming toggle** (entity_types.templ):
   - Visible in the "Customize" hub only when addon is on
   - Checkbox: "Players can claim entities of this type"
   - Shown on both the type card (quick-edit) and the Add-Category form
   - Defaults to true when addon is on; persists via the service's
     `claimable` field on create/update

4. **Claim-button gating:** The unclaimed claim banner now requires
   `addon-enabled` AND `claimable`. Toggling the addon off hides the
   button (and the roster); re-enable to see it again.

**Handler changes:** `Show`, `Index`, `CreateEntityType`, `UpdateEntityTypeAPI`
all gate claiming UI on the addon and type claimability.

**Model changes:** `ClaimRoster` carrier struct for dashboard rendering.

**New files:** `claim_banner.templ`, `category_owner_roster.templ`,
`claim_overview.go` (helpers: `ownerDisplayNames`, `resolveOwnerName`),
`claim_overview_test.go` (264 lines of unit tests).

**Modified files:** `show.templ`, `category_dashboard.templ`, `entity_types.templ`,
`handler.go`, `model.go`, `service.go`, `repository.go` (scan-path updates),

### Audit Actions

Two new audit actions in `internal/plugins/audit/model.go`:

```go
ActionEntityClaimed = "entity.claimed"        // Player claims ownership
ActionEntityOwnerChanged = "entity.owner_changed"  // GM reassigns or clears
```

Both are rendered in the activity feed with human-readable labels and
timeline colors. Details for owner_changed include `new_owner_id`,
`new_owner_name`, or `cleared: true`.

### API Endpoints

**POST /campaigns/:id/entities/:eid/claim** (Player+)
- Assigns the calling user as the entity owner
- Idempotent: if the caller already owns it, returns 200 with the entity
- 409 Conflict: if owned by a different player
- 400 Bad Request: if the entity type isn't claimable

**PUT /campaigns/:id/entities/:eid/owner** (Scribe+)
- Request body: `{"owner_user_id": "user-uuid-or-null"}`
- Reassigns the entity to a different player or clears ownership
- Membership validation: owner_user_id must be a campaign member
- No-op if the new value matches current; 200 OK either way

### Design Rationale

See `.ai/decisions.md` (ADR-039) for the full rationale. Key points:
- **Owner-toggleable addon:** Campaigns can opt-in to the feature; existing
  campaigns default to off (no surprises).
- **Per-type claimable flag:** Not all character-shaped types need claiming
  (e.g., NPCs, templates); the Owner can toggle it per type.
- **"Player Character" sub-type + legacy heuristic fallback:** Both paths
  to claiming are supported — new campaigns can use the dedicated sub-type,
  existing campaigns keep claiming on their "Character" type.
- **Claimable-by-default when addon is on:** Removing the toggle every time
  someone enables the addon would be annoying; defaulting to true reflects
  "I want the feature" → "I want my character types to be claimable."

## `character_surface` block — the player-character "big widget"

The dynamic surface is now a real **layout block** (`Type: "character_surface"`,
`Contexts: ["template"]`, `Singleton`), registered in `block_registry_core.go`.
It is the **default page layout for player-character types**: `CreateEntityType`
gives `isPlayerCharacterType(...)` types `CharacterLayout()` (the surface block +
permissions) instead of `DefaultLayout()` (`service.go`, `model.go:CharacterLayout`).
Because it's a normal registered block, owners customize it in the layout editor
like any other (the palette is registry-driven).

- **Server** (`character_surface.go`, `character_surface_block.templ`):
  `blockCharacterSurface` emits `<div data-widget="dynamic-surface">` with an
  inline `<script type="application/json" data-surface-schema>` built by
  `CharacterSurfaceSchemaJSON(entity, entityType, campaignID, csrf, canEdit)` —
  the entity's real data **seeded** (zero extra fetch). `json.Marshal`
  HTML-escapes it, so it's safe inside the script tag (tested).
- **Client** (`static/js/widgets/character_surface.js`, loaded globally after
  `dynamic_surface.js`): registers the generic box renderers `char-header`
  (portrait + type), `char-attributes` (fields grouped by section), `char-entry`.
  A System (Draw Steel) can override these by re-registering the same names.
- **Security:** the description box does NOT inline `EntryHTML` (that would
  bypass role-aware secret stripping). It mounts the **same `editor` widget** the
  standard `entry` block uses (seeded endpoint + csrf + `canEdit`), so secrets and
  edit-gating are identical.
- **Frame fix it surfaced:** `Provider.push` now sets `_promise` so a seeded
  provider isn't clobbered to `null` by the `load()` that `mount()` always calls
  (`dynamic_surface.js`).
- **Auto-premake + migration on enable:** `EnsurePlayerCharacterType(campaignID)`
  idempotently ensures **one** claimable player-character type exists **nested
  under the default "Characters" category** (`ParentTypeID` → the seeded
  `character` type) — the shape PC-claim always intended (ADR-039 / cordinator
  `2026-06-19-pc-claim-design §3`). Three cases (matched by walking the campaign's
  types): (1) **migrate** a stray top-level PC type (preset `player_character` /
  slug `player-character`) an earlier build premade — re-parent it via
  `UpdateEntityType` (changes only the parent pointer, so claimed characters + the
  layout are preserved); (2) **nest a system's own character type** — when a
  system already created its claimable character type (e.g. `drawsteel-character`,
  detected generically via `isClaimableType`, never by system name), re-parent it
  under "Characters" so it reads as the PC sub-category. This is **minimal-touch**:
  **no rename** (the system's terminology is preserved — modularity) and **no
  field copy** (the type already carries the system's fields and already renders
  via its **own slug renderer**, so binding by preset is not required here). No
  generic type is created; (3) else **create** the generic "Player Characters" type
  (`CharacterLayout()` + claimable) nested (top-level as a fallback when no
  `character` category exists) — the system-less fallback. Wired from
  `addons.EnableForCampaign` → `PresetApplier.ApplyAddonEnableEffects`, and replayed
  by the startup backfill (`internal/app/backfill.go`) — so **deploying heals
  production in place**. The boot path only nests/creates (idempotent,
  non-destructive); it never moves or deletes entity data. A re-parented sub-type
  leaves the sidebar top level and is reached via the "Characters" parent (the
  Lore→Timelines sub-category pattern). Tests: `ensure_pc_type_test.go`.
- **Owner-driven duplicate reconciliation (`MergeDuplicatePlayerCharacterType`):**
  the boot path is deliberately non-destructive, so a campaign can end up with
  BOTH a generic `player_character` type (holding the characters, old layout) AND
  a system character type (e.g. `drawsteel-character`, the real sheet, empty) — the
  enable-ordering artifact. The one-time guarded migration `000030` auto-merges the
  unambiguous case on deploy and is **RETAINED permanently** (it is applied in prod;
  deleting an applied migration crash-loops boot — see ADR-044). An **additive**
  owner-triggered reconciliation handles what the migration skips (ambiguous
  `COUNT<>1`, or duplicates arising later), surfaced by the player-character
  SetupProvider on the extension settings page (see `addons/.ai.md` §"Setup Provider
  framework"). `MergeDuplicatePlayerCharacterType`
  (service) picks the unambiguous (generic → system) pair — classified by
  `preset_category` + `slug`/`is_default` only, NO system names, so any pack's
  duplicate is handled — asserts exactly one source AND one target (human-readable
  `apperror` otherwise), and calls the repo's `MoveEntitiesAndDeleteType` (one tx:
  reassign `entity_type_id` so claims, which reference `entities(id)`, follow,
  then a `NOT EXISTS`-guarded DELETE of the emptied generic — `entities.entity_type_id`
  is ON DELETE CASCADE, so emptying first is mandatory). Idempotent (after the
  first merge the generic is gone → nothing-to-merge success). A read-only
  `PlayerCharacterSetupSnapshot` powers the page's checks. Tests:
  `merge_duplicate_pc_test.go`, `pc_setup_snapshot_test.go`.
- **Field reconcile in place (`ReconcileEntityTypeFields`, WS-5 / PC-PRESET-FIELDS):**
  a system pack can declare new character fields (backstory, abilities…) *after* a
  campaign already created its type. Recreating the type would orphan every claimed
  hero, so the reconciler **additively** backfills instead: the pure `mergeNewFields`
  helper appends any declared field whose `Key` is absent (never removes, reorders,
  or overwrites — so a user-edited field wins, and a second run adds nothing →
  idempotent), then `ReconcileEntityTypeFields` persists via `UpdateEntityType`
  (name unchanged → slug stable). `app/preset_applier.go`'s `ApplySystemPresets`
  drives it: it indexes existing types by preset-category then name and upgrades the
  match in place, else creates. Trigger is the existing system-enable path (fires on
  package install/update). Type schema only — never entity data. Tests:
  `reconcile_fields_test.go`.
- **Single-owner duplicate guard (`CreateEntityType`):** because Chronicle allows
  **one type per preset-category**, the addon owns the player-character category.
  When an Owner manually requests a PC type (preset `player_character` /
  slug `player-character`) and one already exists in the campaign, `CreateEntityType`
  rejects it with a `Conflict` (*"a 'Player Characters' category already exists —
  it is provided by the Player Character addon; you don't need to create another"*).
  Generic across systems (no system names); the premake is unaffected since it only
  runs when none exists. This sits on top of the existing addon gate (which still
  requires the `player-character-claiming` addon to create the first one).

## Notes

- The entity profile page composes multiple widgets: title, editor, tags,
  attributes, relations. The page itself is a Templ template that provides mount points.
- Entity type customization is fully implemented with a visual config UI.
- Rich text editor (TipTap) is integrated via the editor widget with view/edit

- Attributes widget mounts with `data-widget="attributes"` for inline field editing.
- Tags and relations widgets also mount on entity profile pages.
- **Footgun — entity_type reads share ONE scan path.** Every `entity_types`
  SELECT routes through the `entityTypeColumns` const + `scanEntityType(rowScanner)`
  helper in repository.go (FindByID/FindBySlug/ListByCampaign/ListChildTypes/
  ListByPresetCategory/ListAll). The scan binds *by position*, so adding a column
  means editing the const and the helper in exactly one place — never copy a
  bespoke SELECT/Scan pair back into a single method (that reintroduces the
  six-way drift this consolidated, and a mismatch panics at runtime, not build).
  `TestEntityTypeRepository_Integration` (run via `make test-int`) is the live-DB
  guard that proves the column/scan order against a real MariaDB.
