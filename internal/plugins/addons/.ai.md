# Addons Plugin

## For humans

Manages the extension framework — installable addons (modules, widgets, integrations) with per-campaign enable/disable controls. Admin manages the global addon registry; campaign owners toggle addons per campaign. PC-Claim addon is the canonical current example: the `player-character-claiming` addon gates the claim UI and Foundry actor-sync Stage 4.

## For AI sessions

### Key files
- `service.go` — `GetEnabledAddons(campaignID)`, `IsEnabled(campaignID, slug)` — the primary seam other plugins use
- `model.go` — `Addon`, `CampaignAddon`, `AddonCategory` / `AddonStatus`
- `handler.go` + `routes.go` — admin CRUD + per-campaign toggle endpoints

### Cross-plugin usage pattern
Other plugins call `addonService.IsEnabled(ctx, campaignID, "player-character-claiming")` (or similar) to gate addon-dependent features. Inject via `handler.SetAddonService(addonService)` at wiring time. Do NOT import addons from plugin service.go — use the injected interface.

## Architecture
Standard plugin: handler → service → repository → MariaDB.

### Files
- `model.go` — Domain types: Addon, CampaignAddon, AddonCategory/Status, DTOs.
- `repository.go` — SQL CRUD for addons + campaign_addons tables.
- `service.go` — Business logic: validation, status checks, campaign enable/disable.
- `handler.go` — HTTP handlers for admin pages + campaign-scoped toggle API.
- `routes.go` — Route registration: admin routes + campaign-scoped routes.
- `admin_addons.templ` — Admin addon management page.
- `campaign_addons.templ` — Per-campaign addon toggle page.

### Database Tables
- `addons` — Global addon registry (id, slug, name, category, status, icon, etc.)
- `campaign_addons` — Per-campaign addon state (campaign_id, addon_id, enabled, config_json)

### Routes

**Admin (site admin only):**
- `GET /admin/addons` — Addon management page
- `POST /admin/addons` — Register new addon
- `PUT /admin/addons/:addonID/status` — Change addon status
- `DELETE /admin/addons/:addonID` — Delete addon

**Campaign (campaign owner only):**
- `GET /campaigns/:id/addons/settings` — Campaign addon settings page
- `GET /campaigns/:id/addons` — JSON list of addons with campaign state
- `PUT /campaigns/:id/addons/:addonID/toggle` — Enable/disable addon

## Dependencies
- `campaigns` — For campaign context middleware and role checks.
- `auth` — For authentication and admin middleware.
- `apperror` — For domain error types.

### Middleware
- `RequireAddon(svc, slug)` — Route-level middleware that checks if an addon is
  enabled for the current campaign. Returns 404 when disabled. Applied to calendar,
  maps, sessions, timeline, and media-gallery route groups.
- `installedAddons` map in `service.go` — Only addons with backing code can be
  activated by admins or enabled by campaign owners.

## Key Decisions
- Addons start as "planned" when created, must be set to "active" to be enableable.
- Only addons listed in `installedAddons` can be activated (prevents enabling stubs).
- Disabling an addon doesn't delete config — re-enabling preserves previous settings.
- Campaign addon config stored as JSON in `campaign_addons.config_json`.
