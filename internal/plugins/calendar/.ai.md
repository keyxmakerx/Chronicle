# Calendar Plugin

Two-mode calendar system for campaigns. **Fantasy mode**: arbitrary months, weekdays,
moons (with phase calculations), seasons (with wrap-around date ranges), leap years,
configurable time systems, weather state, zodiac/elemental cycles, festivals.
**Real-life mode**: Gregorian calendar synced to wall clock, user timezone-aware.
Both modes support events (single-day, multi-day, recurring with interval/end date,
with optional start/end times, color, icon, all_day). One calendar per campaign.

## Architecture

- **model.go**: Calendar, Month, Weekday, Moon, Season, Event structs + DTOs.
  `ModeFantasy`/`ModeRealLife` constants. `IsRealLife()` mode check.
  `MoonPhase()` / `MoonPhaseName()` for lunar cycle math.
  `IsLeapYear()`, `YearLengthForYear()`, `MonthDays()` for date arithmetic.
  `ContainsDate()` on Season with wrap-around support.
  `FormatCurrentTime()` on Calendar for HH:MM display.
  `HasTime()`, `FormatTime()`, `FormatEndTime()`, `FormatTimeRange()` on Event.
  `HasRichText()`, `PlainDescription()` on Event for rich text support.
- **repository.go**: SQL CRUD for calendar + sub-resources. Set* methods replace
  all records (bulk update). Event listing with entity JOIN, visibility filtering,
  recurring event support. Events sort by date then time (COALESCE for null times).
- **service.go**: One-calendar-per-campaign enforcement, validation, date advancement
  with month/year rollover (leap-year-aware), time advancement with minute→hour→day
  rollover, CRUD orchestration. Per-user visibility filtering via `filterEventsByUser()`
  and `canUserView()`. Input validation for time system values (positive integers),
  event names (max 255), event dates (month/day >= 1), visibility rules JSON.
- **handler.go**: HTTP handlers for monthly grid, timeline, settings, events, setup,
  advance date/time, import/export. HTMX detection for fragment vs full page. Mode-aware
  CreateCalendar seeds Gregorian months/weekdays for real-life mode.
- **routes.go**: Route registration with role middleware.
- **export.go**: Chronicle native JSON export format (`chronicle-calendar-v1`).
  `ChronicleExport` envelope with calendar config + optional events. `BuildExport()`
  creates export from a fully-loaded Calendar + events.
- **import.go**: Auto-detection and parsing of 4 calendar JSON formats:
  Chronicle native, Simple Calendar (Foundry VTT v1+v2), Calendaria (Foundry VTT),
  and Fantasy-Calendar.com. `DetectAndParse()` inspects top-level JSON keys to
  identify format, then delegates to format-specific parsers. Handles 0-indexed vs
  1-indexed conversion, localization key stripping, day-of-year→month+day conversion,
  and {values: {...}} nesting patterns. Returns `ImportResult` with normalized months,
  weekdays, moons, seasons, eras, and settings.
- **calendar.templ**: Monthly grid, timeline view, event modal (create/edit with
  TipTap rich text editor + @mentions, optional time picker), upcoming events fragment,
  entity-events fragment, setup page with mode chooser (Real Life / Custom Fantasy /
  Import from File). Timeline renders rich HTML descriptions with `templ.Raw()`.
- **calendar_settings.templ**: settings page tabs (General, Months, Weekdays,
  Moons, Seasons, Eras, Categories) with Alpine.js list management. General tab
  includes time system configuration (hours/day, minutes/hour, seconds/minute) and
  current time. Seasons tab includes weather effect field per season. Eras tab
  manages named time periods with year ranges. Real-life mode hides Months /
  Weekdays / Eras / Weather / Cycles / Festivals tabs and date/time/leap-year fields.
  Export button in settings header.
- **calendar_settings_wcf.templ**: Weather / Cycles / Festivals tabs (added in
  C-CAL-WCF-UI, 2026-05-19). Wire missing GET-via-eager-load + PUT pairs onto
  the existing settings page. Weather tab edits the single calendar_weather row
  (PresetID + temperature + wind + precipitation + zone + description). Cycles
  tab manages named periodic cycles + their entries (Tyr's Endlean / Seofean
  shape). Festivals tab manages fixed calendar entries (month+day OR after_month
  intercalary). All three forms render the wire-contract `{error, message,
  category}` error shape inline via respondSettingsError (see settings_errors.go).

## Data Model

- `calendars` — id, campaign_id, mode, name, description, epoch_name,
  current_year/month/day, hours_per_day, minutes_per_hour, seconds_per_minute,
  current_hour, current_minute, leap_year_every, leap_year_offset
- `calendar_months` — id, calendar_id, name, days, sort_order, is_intercalary,
  leap_year_days
- `calendar_weekdays` — id, calendar_id, name, sort_order
- `calendar_moons` — id, calendar_id, name, cycle_days, phase_offset, color
- `calendar_seasons` — id, calendar_id, name, start_month/day, end_month/day,
  description, color, weather_effect
- `calendar_eras` — id, calendar_id, name, start_year, end_year (nullable=ongoing),
  description, color, sort_order
- `calendar_events` — id, calendar_id, entity_id (FK), name, description (ProseMirror
  JSON for rich text, plain text for legacy), description_html (pre-rendered sanitized
  HTML), year/month/day, start_hour, start_minute, end_year/end_month/end_day, end_hour,
  end_minute, is_recurring, recurrence_type, visibility, visibility_rules (JSON,
  per-user allowed/denied lists, migration 000037), category, created_by

## Time System

Calendars have a configurable time system with three parameters:
- **hours_per_day** (default 24) — How many hours in one day
- **minutes_per_hour** (default 60) — How many minutes in one hour
- **seconds_per_minute** (default 60) — How many seconds in one minute

Current time is tracked via `current_hour` and `current_minute`. The `AdvanceTime()`
service method handles rollover: minutes overflow into hours, hours overflow into
days (which then roll over months/years via the same logic as `AdvanceDate()`).

Events optionally have `start_hour/start_minute` and `end_hour/end_minute` (all
nullable). When set, times display on grid chips, timeline, and entity event lists.

## Calendar Modes

Two modes controlled by `calendars.mode`:

- **Fantasy** (`mode='fantasy'`, default): Full customization — arbitrary months,
  weekdays, time units, leap years. All settings tabs visible. Setup seeds generic
  12×30-day months and 7 weekdays.
- **Real-Life** (`mode='reallife'`): Gregorian calendar synced to wall clock.
  Setup seeds correct Gregorian months (Jan 31, Feb 28 +1 leap, etc.), weekdays
  (Sunday–Saturday), 24/60/60 time, leap year every 4. Settings page hides Months,
  Weekdays tabs and date/time/time-system/leap-year fields (Gregorian is fixed).
  Current date/time derived from `time.Now()` + user's IANA timezone.

User timezone stored in `users.timezone` (nullable, default UTC). Managed via
`GET /account` settings page.

## Rich Text Event Descriptions

Event descriptions use the same dual-storage pattern as entity entries:
- **`description`**: ProseMirror JSON (from TipTap `getJSON()`) — used for re-editing
- **`description_html`**: Pre-rendered sanitized HTML (from TipTap `getHTML()` + `sanitize.HTML()`) — used for display

The event create/edit modal embeds a compact TipTap editor instance (no headings,
just inline formatting + @mentions). The editor is initialized lazily on first modal
open. On form submit, both JSON and HTML are sent to the backend.

Legacy events with plain text descriptions (created before Sprint 10) are handled
gracefully: `HasRichText()` returns false, `PlainDescription()` returns the text as-is.
When editing a legacy event, the plain text is loaded into the TipTap editor as a paragraph.

Timeline view renders `description_html` via `templ.Raw()` for rich text events, or
falls back to plain text `<p>` for legacy events. Tooltips use `PlainDescription()`
which returns empty for rich text events (ProseMirror JSON is not user-readable).

## Key Design Decisions

- **One calendar per campaign** (unlike maps' many-per-campaign). Enforced in
  service CreateCalendar().
- **Bulk sub-resource updates**: SetMonths/SetWeekdays/SetMoons/SetSeasons/SetEras
  replace all records rather than individual CRUD — simpler client-side serialization.
- **Percentage-free date system**: Month days and leap year days are integers.
  Year length calculated by summing month days + per-month leap extras.
- **Moon phases**: Continuous 0.0–1.0 value from `cycle_days` and `phase_offset`.
  Mapped to 8 named phases (New, Waxing Crescent, etc.).
- **Season wrap-around**: `ContainsDate()` handles seasons crossing year boundary
  (e.g., Winter from month 11 to month 2).
- **Event categories**: holiday, battle, quest, birthday, festival, travel, custom.
  Each has an icon in the UI.
- **Dual views**: Monthly grid (default) and timeline (chronological, year-grouped).
- **Per-user visibility**: Events support `visibility_rules` JSON column (migration
  000037) with `allowed_users` (whitelist) and `denied_users` (blacklist). Owners
  always see all events. Service-layer `filterEventsByUser()` applies rules after
  role-based filtering. UI in event modal (Owner-only) with Alpine.js member checkboxes.

## Import/Export

Calendar supports import from 4 external formats and export in Chronicle's native format.

### Supported Import Formats

| Format | Detection | Key Mapping |
|--------|-----------|-------------|
| **Chronicle** | `"format": "chronicle-calendar-v1"` | Native round-trip |
| **Simple Calendar** (Foundry VTT) | `"calendar"` key (v1) or `"exportVersion"` + `"calendars"` (v2) | `numberOfDays`→Days, `hoursInDay`→HoursPerDay, 0→1 indexed |
| **Calendaria** (Foundry VTT) | `"days.hoursPerDay"` or months-as-object | `days`/`leapDays`, day-of-year seasons, localization key stripping |
| **Fantasy-Calendar.com** | `"static_data"` + `"dynamic_data"` | `timespans`→Months, `global_week`→Weekdays, `cycle`→CycleDays |

### Export

`GET /campaigns/:id/calendar/export?events=true` returns JSON in Chronicle's native
format with all sub-resources + optional events. Settings page has download button.

### Import Flows

1. **Setup import** (`POST /calendar/import-setup`): Upload file during calendar
   creation. Creates new calendar + applies import + auto-enables addon.
2. **Existing calendar import** (`POST /calendar/import`): Replaces all sub-resources
   on an existing calendar. Destructive operation (months, weekdays, moons, seasons,
   eras all replaced).
3. **Preview** (`POST /calendar/import/preview`): Returns parsed preview without applying.

### Sync API

Import/export also available via REST API:
- `GET /api/v1/campaigns/:id/calendar/export` (read permission)
- `POST /api/v1/campaigns/:id/calendar/import` (write permission)

## Audit Log Taxonomy

Calendar mutations emit audit events at the HTTP handler layer (entities-plugin pattern) via `Handler.logCalendarAudit`, wired by `app/routes.go::calendarHandler.SetAuditService(...)`. Emission is fire-and-forget — failures `slog.Warn` but never block the primary write.

Event names (constants in `internal/plugins/audit/model.go`):

- `calendar.created`, `calendar.updated`, `calendar.deleted`
- `calendar.months_set`, `calendar.weekdays_set`, `calendar.moons_set`, `calendar.seasons_set`, `calendar.eras_set`, `calendar.categories_set`
- `calendar.weather_set`, `calendar.cycles_set`, `calendar.festivals_set`
- `calendar.weather_zones_set` — catalog replace (PR #360)
- `calendar.weather_active_zone_changed` — reserved for `SetActiveWeatherZone` direct calls (no current HTTP entry point; service-method only)
- `calendar.event_created`, `calendar.event_updated`, `calendar.event_deleted`, `calendar.event_visibility_changed`
- `calendar.date_advanced`, `calendar.time_advanced`
- `calendar.imported` — full import (file upload or setup-time)

**Counts-only payload discipline:** bulk Set* events log `{count: N}` in `Details`, not the full array. This matches V1-E precedent + keeps audit_log row size bounded for high-cardinality settings (e.g. categories).

**Per-era CRUD + `SetActiveWeatherZone` + `SetEventTierDefinitions`:** these service-layer methods have no HTTP entry point in calendar plugin today (`SetEventTierDefinitions` is exposed by the campaigns plugin and emits `campaign.event_tier_definitions.updated` there). Wave 1 will surface per-era CRUD via the in-card editor; audit emission will follow at that handler.

**Foundry sync (per-campaign-token) writes are NOT audited today.** SyncAPI's `APIHandler` uses per-campaign tokens with no user session; emitting with empty `user_id` would fail the audit-log validator. Operator decision required on actor model (synthetic `foundry-sync` user vs separate sync-audit table vs skip). Tracked as a follow-up.

## Security & Validation

- **IDOR protection**: `requireEventInCampaign()` verifies event's calendar belongs
  to the campaign before allowing modifications.
- **HTML sanitization**: Rich text `description_html` is sanitized via `sanitize.HTML()`
  on both Create and Update paths.
- **Input validation**: Event names (max 255), month/day (>= 1), time system values
  (hours_per_day, minutes_per_hour, seconds_per_minute must be >= 1), visibility
  enum (everyone/dm_only only), visibility_rules JSON validated on all write paths.
- **Visibility filtering**: `canUserView()` applies role-based + per-user JSON rules.
  Malformed JSON fails open with `slog.Warn` (validated on write, so only affects
  manually corrupted data). Owners always see everything.
- **Role enforcement**: Route middleware + CSRF tokens on all state-changing endpoints.

## Known Limitations

- **Recurring events**: Fields stored (`is_recurring`, `recurrence_type`) but no
  server-side expansion logic. Displayed as metadata only, no "edit series" workflow.
- **No event duplication**: Users must manually recreate similar events.
- **No bulk delete**: Events must be deleted individually.
- **ApplyImport not transactional**: Multi-step import (calendar + months + weekdays +
  moons + seasons + eras) is not wrapped in a single transaction. Partial failure
  leaves calendar in inconsistent state. Should wrap in a transaction in a future sprint.

## Entity Ties (C-CAL-ENTITY-TIES-DATA-MODEL, migration 009)

Optional many-to-many ties between **entities** and calendar **events/eras**,
each carrying a **participation role**. The real persistence behind Phase 1.5's
mock attach-entity picker (operator: "an entity can be but does not have to be
tied to events and eras" — both sides optional). **PR1 = data model + service;
the dual-side UI hooks land in PR2.**

- **`participation_role` enum** (`entity_ties.go` `ParticipationRoles`):
  `involved · present · affected · mentioned` — PINNED to Phase 1.5 R3's
  showcase picker (`__calParticipationRoles`); keep identical so the Phase-2
  port swaps mock→real with no translation. Event ties require a role (empty →
  `involved`); era ties are coarser (role nullable). Validated in the service,
  stored as VARCHAR (not a SQL ENUM — easier to extend).
- **migration 009** (`entity_event_links`, `entity_era_links`): placement =
  **option (a)** hard cross-plugin FKs. Trivially safe: `entities` is a CORE
  table (always migrated before plugins), `calendar_events`/`calendar_eras` are
  this plugin's (migration 001). Precedent: `calendar_events.entity_id` already
  hard-FKs `entities(id)`. All FKs `ON DELETE CASCADE` → **cascade-on-delete is
  DB-enforced** (deleting an entity, event, or era drops its links) — no
  service-layer cascade.
- **entity_ties_repository.go**: link/unlink (idempotent upsert) + both-
  direction queries. `EntitiesForEvent`/`EntitiesForEra` JOIN entities +
  entity_types for display (same source the event list uses);
  `EventsForEntity`/`ErasForEntity` reuse the event/era column projections.
- **entity_ties_service.go**: enum validation + delegation. These methods are
  the **cross-plugin surface** the entities plugin (PR2) consumes via an
  interface — never a direct repo import (CLAUDE.md rule 8).
## World-State Model (C-CAL-WORLDSTATE-SERVER-MODEL, migration 008)

Server-side mirror of the showcase's browser-only `worldState` blob (the
synced atmosphere object driving the sky-band + hourglass). The deliverable is
**data + API, not UI** — it unblocks the Phase-2 production port and the
Phase-5b Foundry push.

- **worldstate.go**: the `WorldStateSeed` wire shape (camelCase JSON tags — a
  1:1 mirror of `static/js/cal-almanac.js` ~L464-481, CATALOG Part 8) + the
  pure `AssembleWorldStateSeed(cal, y, m, d, dayWeather, celestials, moonPhases,
  role)` builder. Moon phase/cyclePct/namedPhase computed from the real
  `Moon.MoonPhase` cycle math; named-phase vocab walk falls back to the
  showcase's short procedural names. New domain types: `DayWeather`,
  `CelestialEvent`, `MoonPhaseVocab`, `SpecialDay`.
- **worldstate_repository.go**: reads for the four migration-008 tables
  (`calendar_day_weather`, `calendar_celestial_events`, `calendar_moon_phases`,
  `calendar_special_days`) + `SetDayWeather`/`SetMoodTint` writes. All
  calendar-scoped (moon-phases join through `calendar_moons`).
- **worldstate_service.go**: `BuildWorldStateSeed` (loads + assembles) and
  `SetWorldState` (persists live mood + date/time via `UpdateCalendar`, emits
  `calendar.worldstate.changed`). `timepieceFill`/`timeControl` are ephemeral —
  never persisted; the seed emits `timeControl` with neutral defaults.
- **worldstate_handler.go** + routes: `GET/PUT /campaigns/:id/calendar/world-state`.
  GET = Player+ (GM-only celestial events filtered via `VisibilityRole`); PUT =
  Owner-only (route-level seam for the Phase-3 co-GM grant). No calId — resolves
  the active calendar (or `?calendarId=`). NB: the dispatch sketched an
  `/api/v1/` path but the GET/PUT are the session-auth production surface
  (roles + dm_only only exist on `CampaignContext`); a token-auth Foundry
  variant can layer on the same service later.
- **migration 008** also ALTERs `calendar_moons` (+base_design/tint/phase_source/
  size/orbit_speed — moon-library render params, read via `GetMoons`) and
  `calendars` (+mood_tint_color/intensity). The parity test
  `TestWorldStateSeed_ParityWithJSContract` is the load-bearing guard against
  server↔JS drift.
- **Footgun**: `SetMoons` (DELETE+INSERT replace) does not write the new moon
  render-prop columns, so a moon edit resets them to defaults. No UI authors
  them yet (read-only for now); revisit when a render-prop write path lands.

## Co-DM Capability (C-CAL-COGM-CAPABILITY, Phase 3 / D6)

viewing — **no migration, no new role tier** (reuses `DmGrantIDs` +

## Entity-Page Calendar Embed (C-CAL-ENTITY-PAGE-EMBED, Phase 6 pulled forward)

The `entity_calendar` block — a **template-context** entity-page embed (NOT the
dashboard `calendar_preview` upcoming-events card). On an entity's page it
shows a **compact worldstate band** (reuses the 2a sky scaffold +
`cal-almanac.js` via the `#cal-v2-worldstate` seed blob; rendering-canvas CSS
exemption applies) over **this entity's linked events** (#402 `EventsForEntity`,
each with its `participation_role`), read-only.

- **`EntityCalendarBlock(svc, cc, entityID, userID)`** (`entity_calendar_block.go`)
  is the data-prep: `GetCalendar` → `BuildWorldStateSeed` (#401) → marshal seed;
  `EventsForEntity` (#402) → **dm_only filtered by viewer role** (mirrors
  `filterEventsByUser`/`canUserView`). Returns the `entityCalendarBlockView`
  templ. Graceful: no calendar → empty-but-present; no ties → "No linked
  events" (never blank).
- **Registered in `internal/app/routes.go`** as `entity_calendar`
  (`Contexts:["template"]`, `Addon:"calendar"`, `Singleton:true` — the band
  binds the engine's fixed `#cal-v2-worldstate` id), with `calendarService`
  injected via the renderer closure (the map_editor pattern).
- New type over extending `calendar_preview` (which is a dashboard card) — per
  the dispatch recommendation.
- Band richness is a **compact default** (150px sky band, no hourglass) —
  operator visual-tune (flagged).

**Convergence + polish (C-CAL-EMBED-CONVERGE-POLISH):** `entity_calendar` is now
the ONE entity-page calendar. The old per-entity `calendar` block
(`BlockCalendarEvents`) + its `/calendars/entity-events/:eid` route +
`EntityEventsFragment` handler + the always-on auto-appended events list in
`entities/show.templ` were **deleted** (unused; superseded). Stray stored
`type:"calendar"` blocks drop silently (unregistered). Embed states are
friendly, never raw: no calendar → a **"Create calendar" CTA** to
`/campaigns/:id/calendars` (the existing setup wizard); missing entity/campaign
context → `entityCalendarUnavailable` ("couldn't be loaded") instead of a raw
404.
