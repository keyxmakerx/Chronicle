# Skybox widget

**Files:** `internal/widgets/skybox/skybox.templ` (the Go markup), `static/js/widgets/skybox.js` (the client widget).
**Dispatch:** C-SKYBOX-WIDGET (calendar design-notes idea 5 ‚Äî "a sky lane for
celestial events" ‚Äî signed concept, slice 3 of 6; W6-B this wave).

## What it is

Graduates the animated almanac sky (starfield, sun, moons, weather +
celestial particles ‚Äî everything EXCEPT the hourglass shelf) out of the
calendar plugin's ad-hoc worldState band into a mountable
`data-widget="skybox"` widget: usable in the calendar's own sky pane, on
campaign dashboards, and anywhere `data-widget` works. It is the sky-only
sibling of the existing `worldstate` widget (`static/js/widgets/worldstate.js`
+ `.ai.md`), which is sky **+ hourglass**.

**Engine reuse, not a rewrite.** Like its `worldstate` sibling, this widget
does NOT reimplement sky rendering ‚Äî it reuses `static/js/cal-almanac.js`'s
existing SKY_FX/particle engine verbatim, driving it via
`window.__calSetWorldState`. See "Honest deviation" below for why the actual
particle/spawn code stayed in cal-almanac.js rather than moving files.

## Honest deviation from the dispatch's literal wording

The dispatch asked to "extract the sky-render PORTION of
`static/js/cal-almanac.js`" into this widget. A literal code-move (lifting
`SKY_FX`/`mkMeteors`/the particle engine out of the 4000-line IIFE into
`skybox.js`) was assessed and rejected: cal-almanac.js is a single monolith
with `test/js/harness.mjs` booting the WHOLE file in a Node `vm` sandbox and
several suites (`engine.test.mjs`, `weather.test.mjs`, `timecontrol.test.mjs`,
`tuner.test.mjs`, `css_render_split.test.mjs`, `worldstate.test.mjs`) poke at
its internals directly ‚Äî plus the sky code shares module state (`worldState`,
`VIEW`, `DATA`, the shared particle engine) with hourglass/time-control/demo
code that stays almanac-page-only. A physical split risked exactly the
"entangled... behavior change" the dispatch's own stop-and-flag clause
anticipates. `worldstate.ai.md` already set the precedent for this exact
situation ("the widget does NOT reimplement... ‚Äî it reuses
static/js/cal-almanac.js... Engine reuse, not rebuild") ‚Äî this widget follows
that precedent rather than re-litigating it.

What DID move: the sky becomes independently mountable (the architectural
goal) via this Go widget package + `skybox.js`, and the almanac page's own
`worldStateSkyBandV2` now delegates to it instead of hand-rolling the same
markup ‚Äî so "the almanac page becomes a consumer of the widget" is true at
the template-composition level, even though the rendering engine's *code*
still lives in one file.

## Why a provider singleton (shared with `worldstate`)

Both widgets read `ChronicleWorldState.get(campaignId)`. A page with both a
`skybox` embed and a `worldstate` (hourglass) embed still fetches
`GET /campaigns/:id/calendar/world-state` at most once.

## Footguns

- **Fixed engine id:** the engine binds its FIRST `[data-cal-worldstate]`
  band per page (worldstate.ai.md's footgun #1 applies here too) ‚Äî a page
  showing both `skybox` and `worldstate`/`entity_calendar` renders only the
  first one live; the rest paint their static SSR sky and stay static.
- **Script tags are the caller's job, not `Skybox()`'s.** `Skybox()` renders
  markup only ‚Äî no `<script>`/`<link>` tags ‚Äî because a page composing sky +
  hourglass together (the existing `worldStateBandVariantV2`) needs
  `cal-almanac.js` exactly once, not once per sub-component.
  `init()`/`teardownProd()` in cal-almanac.js already guard double-execution
  (`band.__calInited`), so an accidental double `<script>` tag is a harmless
  no-op, not a real bug class ‚Äî but don't rely on that; follow the existing
  callers' pattern (one script-tag set per composed band).
- **`ElementID` is optional, and that's deliberate.** Leave it `""` when
  your caller already renders its own externally-namespaced seed blob (the
  three delegating callers do ‚Äî `worldStateSkyBandV2`'s `skyboxDataFor`
  never sets it); set it to get a self-contained mount (the skybox
  dashboard/template block does, via `skyboxDataForBlock`). Setting it in
  BOTH places on one page duplicates the blob ‚Äî harmless (the engine only
  reads the first `[data-cal-worldstate]` match) but pick one.
- **First paint vs live updates:** same as `worldstate` ‚Äî the first frame
  comes from the embedded seed blob; `window.__calSetWorldState` is a no-op
  until the engine has booted (fine, the blob already painted).
- **`sunBody` is required.** `Skybox()` takes the sun's CSS-art body as a
  child component (`calendar.CalAlmanacSunBody()` today) rather than owning
  it ‚Äî passing a `nil`/empty component renders an empty sun disc, not a
  crash, but callers should always pass the real body.

## Meteor pacing fix (this dispatch's second half)

The operator-reported "one meteor then minutes of nothing" bug lives in
cal-almanac.js's `mkMeteors`/`effectLayersFor` (NOT in this widget package ‚Äî
see cal-almanac.js's own comments at `mkMeteors` and `LAYER_CACHE` for the
verified root cause and fix). Summary: `effectLayersFor()` used to rebuild
every active SKY_FX layer from scratch on every refeed ‚Äî which fires on
every `timeOfDay` tick during a DM time-advance tween ‚Äî discarding the
meteor closure's live state each time. Fixed by caching layer instances
across refeeds while the active effect set is unchanged. Spawn-rate-tied-to-
intensity and viewport-bound-spawn are pure, unit-tested functions in the
same file (`meteorShowerIntensity`/`meteorShowerProgress`/
`meteorSpawnRate`/`meteorSpawnPoint`, tested in
`test/js/skybox_meteor_pacing.test.mjs`).
