# Skybox widget

**Files:** `internal/widgets/skybox/skybox.templ` (the Go markup), `static/js/widgets/skybox.js` (the client widget).
**Dispatch:** C-SKYBOX-WIDGET (calendar design-notes idea 5 — "a sky lane for
celestial events" — signed concept, slice 3 of 6; W6-B this wave).

## What it is

Graduates the animated almanac sky (starfield, sun, moons, weather +
celestial particles — everything EXCEPT the hourglass shelf) out of the
calendar plugin's ad-hoc worldState band into a mountable
`data-widget="skybox"` widget: usable in the calendar's own sky pane, on
campaign dashboards, and anywhere `data-widget` works. It is the sky-only
sibling of the existing `worldstate` widget (`static/js/widgets/worldstate.js`
+ `.ai.md`), which is sky **+ hourglass**.

## Architecture

```
[Go]  internal/widgets/skybox/skybox.templ
        SkyboxData          — flat, plugin-agnostic rendering payload
        Skybox(data, sunBody templ.Component)
          renders the sky surface scaffold wrapped in data-widget="skybox",
          plus an OPTIONAL hidden worldstate-seed blob (only when
          data.ElementID != "" — see "ElementID is optional" below).
          sunBody is injected (not owned here) so calendar.CalAlmanacSunBody
          stays the ONE source for the sun's CSS-art body (also reused by
          /demo).

[Go]  internal/plugins/calendar/calendar_v2_worldstate.templ
        worldStateSkyBandV2(data CalendarV2ViewData) — the almanac page's
        existing entry point — now builds a skybox.SkyboxData from its own
        ws* helpers and DELEGATES to skybox.Skybox(...). Same output, so
        every existing consumer (the full calendar_v2 band, the dashboard
        "See in action" shelf card, entity_calendar_block.templ,
        entity_worldstate_block.templ) is unchanged — the almanac page is
        now a CONSUMER of the widget, not a bespoke reimplementation.

[Go]  internal/plugins/calendar/entity_skybox_block.go/.templ
        The NEW dashboard/template block (Type: "skybox") — sky only, no
        hourglass, campaign-level (always the campaign's default calendar,
        no per-entity binding — mirrors entity_worldstate's "campaign-level,
        no per-entity data" reasoning, just without the binding machinery
        entity_worldstate needs for its per-host hourglass-calendar choice).
        Registered in internal/app/routes.go next to the other "Calendar
        plugin blocks" (Addon: "calendar").

[client] boot.js → Chronicle.register('skybox')  (static/js/widgets/skybox.js)
   ├─ ChronicleWorldState.get(campaignId)     (worldstate_provider.js — the
   │     SAME singleton the worldstate widget uses; N widgets on one page
   │     share ONE fetch)
   └─ on each provider update → window.__calSetWorldState(seed)  (drives the
        SHARED engine, static/js/cal-almanac.js — see worldstate.ai.md's
        "Engine reuse, not rebuild": this widget follows the identical
        pattern, just without the hourglass surface)
```

**Engine reuse, not a rewrite.** Like its `worldstate` sibling, this widget
does NOT reimplement sky rendering — it reuses `static/js/cal-almanac.js`'s
existing SKY_FX/particle engine verbatim, driving it via
`window.__calSetWorldState`. See "Honest deviation" below for why the actual
particle/spawn code stayed in cal-almanac.js rather than moving files.

## Honest deviation from the dispatch's literal wording

The dispatch asked to "extract the sky-render PORTION of
`static/js/cal-almanac.js`" into this widget. A literal code-move (lifting
`SKY_FX`/`mkMeteors`/the particle engine out of the 4000-line IIFE into
`skybox.js`) was assessed and rejected: cal-almanac.js is a single monolith
with `test/js/harness.mjs` booting the WHOLE file in a Node `vm` sandbox and
several suites (`engine.test.mjs`, `weather.test.mjs`, `timecontrol.test.mjs`,
`tuner.test.mjs`, `css_render_split.test.mjs`, `worldstate.test.mjs`) poke at
its internals directly — plus the sky code shares module state (`worldState`,
`VIEW`, `DATA`, the shared particle engine) with hourglass/time-control/demo
code that stays almanac-page-only. A physical split risked exactly the
"entangled... behavior change" the dispatch's own stop-and-flag clause
anticipates. `worldstate.ai.md` already set the precedent for this exact
situation ("the widget does NOT reimplement... — it reuses
static/js/cal-almanac.js... Engine reuse, not rebuild") — this widget follows
that precedent rather than re-litigating it.

What DID move: the sky becomes independently mountable (the architectural
goal) via this Go widget package + `skybox.js`, and the almanac page's own
`worldStateSkyBandV2` now delegates to it instead of hand-rolling the same
markup — so "the almanac page becomes a consumer of the widget" is true at
the template-composition level, even though the rendering engine's *code*
still lives in one file.

## Why a provider singleton (shared with `worldstate`)

Both widgets read `ChronicleWorldState.get(campaignId)`. A page with both a
`skybox` embed and a `worldstate` (hourglass) embed still fetches
`GET /campaigns/:id/calendar/world-state` at most once.

## Footguns

- **Multi-instance IS supported (C-SKYBOX-MULTI-INSTANCE, fixed 2026-07-18).**
  Previously the engine bound its FIRST `[data-cal-sky]`/`[data-cal-worldstate]`
  band per page only — a page showing two `skybox` mounts (e.g. a dashboard
  block + the calendar sky pane open at once) rendered the second one static
  forever. `cal-almanac.js` now keeps a `SKY_BANDS` array — one canvas-surface
  set + one `layerCache` per live `[data-cal-sky]` root, all still driven by
  the SAME shared `CalParticleEngine` rAF loop and the SAME `worldState` (the
  #543 doctrine: engine reuse, one loop, per-instance layer state). See the
  "LAYER_CACHE"/"SKY_BANDS"/`bindSkyBands`/`paintSkyBands` comments in
  `cal-almanac.js` and `test/js/skybox_multi_instance.test.mjs` for the
  repro + pins (both tick; destroy of one doesn't kill the other;
  reduced-motion freezes every instance identically). `SKY_SURFACE`/
  `SKY_FRONT`/`window.__calSkyEngine` still point at the FIRST bound band
  only, for back-compat with existing single-band callers/tests — reach for
  `SKY_BANDS`/`bandForRoot()` if you need a specific instance's surface.
  A page mixing `skybox` with `worldstate`/`entity_calendar` blocks in the
  SAME layout slot is a separate, still-standing constraint — see
  `worldstate.ai.md`'s footgun #1: that one is a server-side block-registry
  `Singleton` rule (pick one block type per slot), not an engine limitation.
- **Script tags are the caller's job, not `Skybox()`'s.** `Skybox()` renders
  markup only — no `<script>`/`<link>` tags — because a page composing sky +
  hourglass together (the existing `worldStateBandVariantV2`) needs
  `cal-almanac.js` exactly once, not once per sub-component.
  `init()`/`teardownProd()` in cal-almanac.js already guard double-execution
  (`band.__calInited`), so an accidental double `<script>` tag is a harmless
  no-op, not a real bug class — but don't rely on that; follow the existing
  callers' pattern (one script-tag set per composed band).
- **`ElementID` is optional, and that's deliberate.** Leave it `""` when
  your caller already renders its own externally-namespaced seed blob (the
  three delegating callers do — `worldStateSkyBandV2`'s `skyboxDataFor`
  never sets it); set it to get a self-contained mount (the skybox
  dashboard/template block does, via `skyboxDataForBlock`). Setting it in
  BOTH places on one page duplicates the blob — harmless (the engine only
  reads the first `[data-cal-worldstate]` match) but pick one.
- **First paint vs live updates:** same as `worldstate` — the first frame
  comes from the embedded seed blob; `window.__calSetWorldState` is a no-op
  until the engine has booted (fine, the blob already painted).
- **`sunBody` is required.** `Skybox()` takes the sun's CSS-art body as a
  child component (`calendar.CalAlmanacSunBody()` today) rather than owning
  it — passing a `nil`/empty component renders an empty sun disc, not a
  crash, but callers should always pass the real body.

## Meteor pacing fix (this dispatch's second half)

The operator-reported "one meteor then minutes of nothing" bug lives in
cal-almanac.js's `mkMeteors`/`effectLayersFor` (NOT in this widget package —
see cal-almanac.js's own comments at `mkMeteors` and `LAYER_CACHE` for the
verified root cause and fix). Summary: `effectLayersFor()` used to rebuild
every active SKY_FX layer from scratch on every refeed — which fires on
every `timeOfDay` tick during a DM time-advance tween — discarding the
meteor closure's live state each time. Fixed by caching layer instances
across refeeds while the active effect set is unchanged. Spawn-rate-tied-to-
intensity and viewport-bound-spawn are pure, unit-tested functions in the
same file (`meteorShowerIntensity`/`meteorShowerProgress`/
`meteorSpawnRate`/`meteorSpawnPoint`, tested in
`test/js/skybox_meteor_pacing.test.mjs`).
