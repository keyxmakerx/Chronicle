# Entity Notes Widget

## For humans

Player-facing entity-page notes widget. Each note belongs to one author and has an audience setting (`private` / `dm_only` / `dm_scribe` / `everyone` / `custom`) that controls who can read it. The widget mounts on entity show pages; auto-renders via `boot.js`'s `data-widget` convention.

Distinct from the `widgets/notes/` plugin (which is the journal-style note widget); `entity_notes` is attached-to-an-entity scope, where `notes` is its own first-class object.

## For AI sessions

### Key files
- `model.go` — Note + audience-enum types
- `repository.go` — SQL CRUD
- `service.go` — business logic (audience enforcement, bluemonday HTML sanitization on write)
- `handler.go` — HTTP handlers (HTMX fragments + JSON for the widget)
- `routes.go` — `RegisterRoutes(e, h, campaignSvc, authSvc)` mounts everything under `/campaigns/:id/entities/:entityID/notes`
- `entity_notes.templ` — widget UI

### Routes (under `/campaigns/:id/entities/:entityID/notes`)
- `GET /` — list notes (audience-filtered for caller)
- `POST /` — create note
- `PUT /:noteID` — edit note
- `DELETE /:noteID` — delete note

### Wiring
Constructed inline in `internal/app/routes.go` and registered via `entity_notes.RegisterRoutes(...)`. Session-cookie auth + campaign-access gate.

### Dependencies
- `entities` — for entity context resolution (the note's parent)
- `campaigns` — campaign-access middleware
- `auth` — session auth
- `sanitize` — bluemonday HTML sanitization on every write (per Pillar 1 + the security audit's §1.3 inventory)

### Footguns
- Audience-enforcement at the service layer is the canonical check; never trust client-side filters
- Custom-audience notes carry `note_audience_grants` rows; check both the audience enum AND the per-user grants when filtering
