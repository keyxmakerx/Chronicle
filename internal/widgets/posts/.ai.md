# Posts Widget

## For humans

Entity posts widget. Posts are named content sections attached to entities, displayed below the main entry on the entity show page. Each post has its own TipTap-edited rich-text body + optional privacy flag.

Use case: a wiki-style entity (e.g. an NPC) might have "Background", "Combat Stats", and "DM Secrets" posts under it, each independently editable, with the DM-secret post hidden from non-DM members. The main `entry`/`entry_html` field stays the entity's overview; posts are the per-section deep-dives.

## For AI sessions

### Key files
- `model.go` — Post + DTO types; includes `EntryHTML` + `IsPrivate` fields
- `repository.go` — SQL CRUD against the `posts` table
- `service.go` — business logic (HTML sanitization on write via `sanitize.HTML`)
- `handler.go` — HTTP handlers (HTMX fragments + JSON)
- `routes.go` — `RegisterRoutes(e, h, campaignSvc, authSvc)` mounts everything under `/campaigns/:id/entities/:entityID/posts`
- `posts.templ` — widget UI

### Routes (under `/campaigns/:id/entities/:entityID/posts`)
- `GET /` — list posts (privacy-filtered for caller)
- `POST /` — create post
- `PUT /:postID` — edit post
- `DELETE /:postID` — delete post

### Wiring
Constructed inline in `internal/app/routes.go` and registered via `posts.RegisterRoutes(...)`. Session-cookie auth + campaign-access gate.

### Dependencies
- `entities` — for entity context resolution
- `campaigns` — campaign-access middleware
- `auth` — session auth
- `sanitize` — bluemonday HTML sanitization on every write

### Backup/restore
Posts participate in the campaigns export/import system via `internal/app/export_adapters.go` (`postExportAdapter` + `postImportAdapter`). Per the C-SEC audit §2 M-4 + §0.5 D4=(c) decision, the round-trip is lossless — no re-sanitization on the backup egress path.
