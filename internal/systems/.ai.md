# System Framework

## Purpose

The system framework enables game system content packs to be added to Chronicle
with minimal or zero custom Go code. Systems are **read-only** — they serve
reference data (spells, monsters, items, classes, etc.) as browsable pages,
searchable lists, and hover tooltips. They never modify campaign data.

## Architecture

### Two Paths to Create a System

**1. Zero-Code (Generic System)** — Drop files, done:
```
internal/systems/my-system/
  manifest.json    # Categories, fields, metadata
  data/
    spells.json    # ReferenceItem[] per category
    monsters.json
    feats.json
```

The loader auto-discovers the manifest, loads JSON data via `JSONProvider`,
and creates a `GenericModule` with a `GenericTooltipRenderer` that reads
field definitions from the manifest. No Go code needed.

**2. Custom Factory (Override)** — For richer rendering:
```go
// internal/systems/dnd5e/dnd5e.go
func init() {
    systems.RegisterFactory("dnd5e", func(m *systems.ModuleManifest, dataDir string) (systems.Module, error) {
        // Custom tooltip renderer, special formatting, etc.
        return NewDnD5eModule(m, dataDir)
    })
}
```

Register a factory in `init()` and the loader uses it instead of the generic
path. The D&D 5e system does this for category-specific stat-block formatting.

### Discovery & Loading Flow

```
systems.Init("internal/systems")
  └── ModuleLoader.DiscoverAll()
        ├── Scan subdirectories for manifest.json
        ├── Parse + validate each manifest
        ├── For each "available" system:
        │     ├── Factory registered? → factory(manifest, dataDir)
        │     └── No factory? → NewGenericModule(manifest, dataDir)
        └── Sort by name, register in global registry
```

### Key Types

| Type | File | Role |
|------|------|------|
| `Module` | system.go | Interface: Info + DataProvider + TooltipRenderer |
| `ModuleManifest` | manifest.go | JSON manifest with categories, fields, presets |
| `CategoryDef` | manifest.go | Category slug, name, icon, field schema |
| `FieldDef` | manifest.go | Property key, label, type (string/number/list/markdown) |
| `ReferenceItem` | system.go | Universal data shape: name, summary, properties map, tags |
| `DataProvider` | system.go | Interface: List, Get, Search, Categories |
| `TooltipRenderer` | system.go | Interface: RenderTooltip, SupportedCategories |
| `JSONProvider` | json_provider.go | Loads `data/*.json`, auto-discovers categories |
| `GenericModule` | generic_system.go | Module impl using JSONProvider + GenericTooltipRenderer |
| `GenericTooltipRenderer` | generic_tooltip.go | Manifest-driven tooltip: iterates FieldDefs in order |
| `ModuleLoader` | loader.go | Scans directories, discovers manifests, instantiates modules |

### Manifest Schema

```json

  "id": "my-system",
  "name": "My RPG System",
  "description": "Reference data for My RPG",

  "author": "You",
  "license": "CC-BY-4.0",
  "icon": "fa-dice-d20",
  "api_version": "1",
  "status": "available",
  "categories": [
    {
      "slug": "spells",
      "name": "Spells",
      "icon": "fa-magic",
      "fields": [
        { "key": "level", "label": "Spell Level", "type": "number" },
        { "key": "school", "label": "School", "type": "string" },
        { "key": "components", "label": "Components", "type": "string" }
      ]
    }
  ],
  "entity_presets": [
    {
      "slug": "my-character",

      "icon": "fa-user",
      "color": "#4a90d9",
      "fields": [
        { "key": "level", "label": "Level", "type": "number" }
      ]
    }
  ]
}
```

### Data File Format

Each `data/<category>.json` contains an array of `ReferenceItem`:

```json
[
  {
    "id": "fireball",
    "name": "Fireball",
    "summary": "A bright streak of fire explodes in a 20-foot radius",
    "description": "Full spell text here...",
    "properties": {
      "level": 3,
      "school": "Evocation",
      "components": "V, S, M"
    },
    "tags": ["evocation", "fire", "damage"],
    "source": "Core Rulebook"
  }
]
```

The filename determines the category slug (`spells.json` → "spells").

### Tooltip Rendering

**Generic (default):** `GenericTooltipRenderer` iterates the manifest's
`fields` array for the item's category, rendering each field with its
label and value. Only manifest-declared fields are shown, in manifest order.
Extra properties in the data are ignored in tooltips.

**Custom (override):** A module can implement its own `TooltipRenderer`
interface for richer formatting (stat blocks, icons, conditional layout).
The D&D 5e module does this with category-specific property rendering.

### Routes (Shared Handler)

All systems share the same handler (`handler.go`):

| Method | Path | Handler | Description |
|--------|------|---------|-------------|
| GET | /campaigns/:cid/systems | SystemIndex | List enabled systems |
| GET | /campaigns/:cid/systems/:mid | CategoryList | System category cards |
| GET | /campaigns/:cid/systems/:mid/:cat | ItemList | Searchable item list |
| GET | /campaigns/:cid/systems/:mid/:cat/:id | ItemDetail | Full reference page |
| GET | /campaigns/:cid/systems/:mid/search | SearchAPI | JSON search results |
| GET | /campaigns/:cid/systems/:mid/tooltip | TooltipAPI | HTML tooltip fragment |

## System Data Loading

**Bundled data has been removed.** As of 2026-03-18, the `dnd5e/data/`,
`pathfinder2e/`, and `drawsteel/` bundled directories have been deleted.
Systems now load exclusively from the **package manager** via
`LoadAdditionalDir()` at startup. The `Init("internal/systems")` call
scans the bundled directory but finds no manifests — it logs
"no bundled systems found" and proceeds normally.

The **dnd5e custom factory** (`dnd5e/dnd5e.go`) is preserved. When dnd5e
is installed via the package manager, the system loader uses this factory
with the external data directory from `media/packages/systems/dnd5e/`.

**Foundry module** (`foundry-module/`) is also preserved as a fallback
until the external `chronicle-foundry-module` GitHub repo is created.

## How to Add a New Game System

Systems are now created as **external packages** installed via the package
manager. To create a new system:

1. Create a GitHub repo with `manifest.json` + `data/<category>.json`
2. Tag a release (e.g., `v1.0.0`)
3. Register the package in Chronicle's admin UI
4. Install via the package manager — auto-discovered on install

For bundled development/testing, you can still place a manifest + data
under `internal/systems/<id>/` and it will be auto-discovered at startup.

For custom tooltip formatting, add a Go file with `init()` that calls
`systems.RegisterFactory()`.

### Custom Game System Upload

Campaign owners can upload custom game systems as ZIP files:
- `POST /campaigns/:id/systems/upload` — Upload ZIP (manifest.json + data/*.json)
- `DELETE /campaigns/:id/systems/custom` — Remove custom system
- `GET /campaigns/:id/systems/custom` — Get custom system status
- Stored under `media/modules/<campaignID>/`, auto-prefixed with `custom-`
- Bypasses addon check for the uploading campaign

## Auto-Registration as Addons

As of 2026-03-19, game systems are **automatically registered as addons**
at startup. The systems registry exports `AddonInfos()` which returns
addon metadata (slug, name, description, version, icon, author) for all
discovered systems with status "available". The app wiring calls
`addons.RegisterSystemAddon()` for each before `SeedInstalledAddons()`.

This means:
- **No hardcoded addon definitions** for game systems
- Dropping a system folder into `internal/systems/` auto-registers it
- Installing via the package manager auto-registers it
- Campaign owners uploading a ZIP auto-registers it (via CampaignSystemManager)
- The only Go import needed is for systems with **custom Go code** (e.g., dnd5e's custom tooltip renderer)

## Dependencies

- **Uses:** Nothing (read-only, self-contained, no DB access)
- **Used by:** Entity pages (tooltips on @mentions), campaign addon settings, addon registry (auto-registration)
